Release notes 发行说明

WiX v5​ WiX v5

WiX v5 marks the first of our annual releases. We intentionally made WiX v5 highly compatible with WiX v4. Most users can simply switch to WiX v5 with no code changes. (There are some exceptions, naturally.) Here's a list of some notable changes, mostly new features to make your package experience with WiX more pleasant:
WiX v5 标志着我们年度发布的开始。我们有意使 WiX v5 与 WiX v4 高度兼容。大多数用户可以直接切换到 WiX v5 而无需更改代码。（当然，有一些例外。）以下是一些显著变化的列表，主要是一些新功能，使您在使用 WiX 时的打包体验更加愉快：

FeatureDescription
Built-in file harvesting 内置文件收集Include files in your projects using wildcards. No Heat required.
使用通配符将文件包含在您的项目中。无需 Heat。
Naked files 裸文件Simplify component authoring that contains only files.
简化仅包含文件的组件编写。
Default major upgrades 默认主要升级Get a typical major upgrade with no XML.
获取无 XML 的典型主要升级。
Default installation folder
默认安装文件夹		Get a typical installation directory with no XML.
获取没有 XML 的典型安装目录。
Default feature 默认功能Get a typical feature with no XML.
获取一个没有 XML 的典型功能。
Modern Windows Firewall support
现代 Windows 防火墙支持		WixToolset.Firewall.wixext supports more Firewall features. Might require minor authoring changes.
WixToolset.Firewall.wixext 支持更多防火墙功能。可能需要进行一些小的编写更改。
Virtual and overridable symbols
虚拟和可重写的符号		Overridability can be applied to many things now. Might require minor authoring changes.
现在可以将可重写性应用于许多事物。可能需要进行一些小的编写更改。
Out-of-process bootstrapper applications
进程外引导程序应用程序		BAs are no longer hosted in the Burn process, for better reliability, security, and compatibility. Custom BAs will require source-code changes.
BA 不再托管在 Burn 过程中，以提高可靠性、安全性和兼容性。自定义 BA 需要进行源代码更改。

Read more about them in WiX v5 for WiX v4 users.
在 WiX v5 for WiX v4 users 中阅读更多相关信息。

WiX v5 releases​ WiX v5 发布

  • WiX v5.0.0 was released on 5-April-2024, the 20th anniversary of the first open-source release of WiX. It contains fixes for a small number of bugs.
    WiX v5.0.0 于 2024 年 4 月 5 日发布，这是 WiX 第一次开源发布的 20 周年纪念日。它包含少量错误修复。
  • WiX v5.0.0-rc.2 was released on 22-March-2024. It contains fixes for a small number of bugs and for security vulnerabilities that FireGiant also fixed in WiX v3.14.1 and WiX v4.0.5. For details, see the FireGiant blog post, the first security advisory, and the second security advisory.
    WiX v5.0.0-rc.2 于 2024 年 3 月 22 日发布。它包含少量错误修复和 FireGiant 也在 WiX v3.14.1 和 WiX v4.0.5 中修复的安全漏洞。详情请参阅 FireGiant 博客文章、第一份安全公告和第二份安全公告。
  • WiX v5.0.0-rc.1 was released on 8-March-2024.
    WiX v5.0.0-rc.1 于 2024 年 3 月 8 日发布。

WiX v4.0.5​ WiX v4.0.5

WiX v4.0.5, released on Friday, 22-Mar-2024, contains fixes for two security vulnerabilities. All versions of WiX are affected by this vulnerability. We recommending upgrading to this version as soon as possible. For details, see the FireGiant blog post, the first security advisory, and the second security advisory.
WiX v4.0.5 于 2024 年 3 月 22 日星期五发布，包含两个安全漏洞的修复程序。所有版本的 WiX 都受此漏洞影响。我们建议尽快升级到此版本。详情请参阅 FireGiant 博客文章、第一份安全公告和第二份安全公告。

Platforms​ 平台

  • Arm64 is supported in the core toolset, extensions, and Burn.
    Arm64 在核心工具集、扩展和 Burn 中受支持。
  • WiX extensions include platform-specific custom actions for the platform of the package being built. For example, an Arm64 package contains only Arm64 custom actions and doesn't rely on x86 emulation.
    WiX 扩展包括针对正在构建的软件包平台的特定平台自定义操作。例如，Arm64 软件包仅包含 Arm64 自定义操作，不依赖于 x86 仿真。

Build tools​ 构建工具

  • WiX v4 doesn't have to be installed on every dev machine and build image like WiX v3. Instead, WiX v4 follows the modern .NET model of using NuGet to deliver tools.
    WiX v4 不必像 WiX v3 那样在每台开发机器和构建镜像上安装。相反，WiX v4 遵循现代.NET 模型，使用 NuGet 来交付工具。
    • WiX v4 MSBuild projects are SDK-style projects; MSBuild and NuGet work together to bring down the WiX v4 MSBuild SDK NuGet package.
      WiX v4 MSBuild 项目是 SDK 风格的项目；MSBuild 和 NuGet 一起工作以下载 WiX v4 MSBuild SDK NuGet 包。
    • Both .NET Framework MSBuild and dotnet build are supported.
      支持 .NET Framework MSBuild 和 dotnet build 。
    • To build packages from the command line, WiX v4 is available as a .NET tool, which is also a NuGet package.
      要从命令行构建包，WiX v4 可作为 .NET 工具使用，它也是一个 NuGet 包。
    • WiX extensions are delivered as NuGet packages, which are usable from both MSBuild projects via PackageReference and the WiX .NET tool.
      WiX 扩展作为 NuGet 包提供，可以通过 PackageReference 从 MSBuild 项目和 WiX .NET 工具中使用。
  • For command-line afficionados, most executables have been merged into a single wix.exe tool with commands. For example, in WiX v3, you might build by calling Candle.exe one or more times to compile your authoring and then calling Light.exe to link and bind the compiled authoring into an .msi package. In WiX v4 using wix.exe, that's one command: wix build -o product.msi product.wxs.
    对于命令行爱好者，大多数可执行文件已合并为一个带有命令的 wix.exe 工具。例如，在 WiX v3 中，您可能会多次调用 Candle.exe 来编译您的创作，然后调用 Light.exe 将编译的创作链接并绑定到 .msi 包中。在 WiX v4 中使用 wix.exe ，这是一条命令： wix build -o product.msi product.wxs 。
  • Building patches is much easier (one command!) and can use MSI packages as the source of target and updated files.
    构建补丁变得更加容易（只需一个命令！），并且可以使用 MSI 包作为目标和更新文件的来源。
  • The WiX extensibility model and pipeline integration has been dramatically enhanced.
    WiX 的可扩展性模型和管道集成得到了极大增强。
  • The WiX language has been further simplified. For example:
    WiX 语言进一步简化。例如：
    • The Package element combines what was two elements (Product and Package) in WiX v3 .
      Package 元素结合了 WiX v3 中的两个元素（ Product 和 Package ）。
    • The StandardDirectory element simplifies the use of standard Windows Installer directories.
      StandardDirectory 元素简化了标准 Windows Installer 目录的使用。
    • The Subdirectory attribute on, for example, the Component element, lets you create subdirectories without nested Directory elements.
      例如， Component 元素上的 Subdirectory 属性允许您在不嵌套 Directory 元素的情况下创建子目录。
    • WiX supplies a default MediaTemplate element if you don't specify one in your authoring.
      如果您在编写中没有指定，WiX 会提供一个默认的 MediaTemplate 元素。
  • WiX warns when mixing authoring meant for MSI packages in bundles and vice versa.
    WiX 警告在捆绑包中混合用于 MSI 包的创作，反之亦然。

Burn, bundles, and bootstrapper applications​
Burn、捆绑包和引导程序应用程序

  • The Burn engine is platform-specific, so you can build an x64 bundle that contains only x64 code and doesn't rely on WoW64.
    Burn 引擎是特定于平台的，因此您可以构建仅包含 x64 代码且不依赖于 WoW64 的 x64 捆绑包。
  • .NET 6 and later are supported platforms for writing managed-code bootstrapper applications. .NET Framework is also supported.
    .NET 6 及更高版本是编写托管代码引导程序应用程序的支持平台。也支持.NET Framework。
  • ThmUtil, the native-code UI library used by the WixStdBA bootstrapper application, supports new controls and authored conditions and actions that let themes add functionality without having to write custom C++ code. For details, see Thmutil schema.
    ThmUtil，WixStdBA 引导程序应用程序使用的本机代码 UI library，支持新的控件和编写的条件和操作，使主题能够添加功能而无需编写自定义 C++代码。有关详细信息，请参阅 Thmutil 架构。
  • ThmUtil (and therefore WixStdBA) supports high DPI display settings.
    ThmUtil（因此 WixStdBA）支持高 DPI 显示设置。
  • The new WixInternalUIBootstrapperApplication BA supports showing only the internal or embedded UI of an MSI package.
    新的 WixInternalUIBootstrapperApplication BA 支持仅显示 MSI 包的内部或嵌入式 UI。
  • Burn support other bundles in the chain via BundlePackage to automatically handle detection and uninstall command lines. Likewise, the ArpEntry element provides the same functionality for arbitrary executable packages in the chain.
    Burn 通过 BundlePackage 支持链中的其他包，以自动处理检测和卸载命令行。同样， ArpEntry 元素为链中的任意可执行包提供相同的功能。
  • Burn supports SemVer-style versions.
    Burn 支持 SemVer 风格的版本。
  • WixStdBA supports bundle update feeds as specified in an Update element.
    WixStdBA 支持按照 Update 元素指定的捆绑更新源。
  • Burn now upgrades bundles with the same version numbers.
    Burn 现在升级具有相同版本号的捆绑包。
  • When a user requires elevation to restart (common on Windows Server machines), Burn handles restart through the elevated engine.
    当用户需要提升权限以重新启动时（在 Windows Server 机器上很常见），Burn 通过提升的引擎处理重新启动。
  • Custom bootstrapper applications can change the REINSTALLMODE used when applying MSI packages.
    自定义引导程序应用程序可以更改应用 MSI 包时使用的 REINSTALLMODE 。
  • The Burn policy registry value EngineWorkingDirectory in HKLM\Software\Policies\Wix\Burn is a string specifying a working folder root directory for elevated bundles when the default of C:\Windows\Temp is blocked by security policy.
    Burn 策略注册表值 EngineWorkingDirectory 在 HKLM\Software\Policies\Wix\Burn 中是一个字符串，指定了一个工作文件夹 root 目录，当默认的 C:\Windows\Temp 被安全策略阻止时使用。
  • Burn no longer appears in the Apps & Features (ARP) list if a bundle has a pre-req that causes a reboot and the user cancels after the reboot.
    如果捆绑包有一个导致重启的前提条件，并且用户在重启后取消，Burn 将不再出现在应用和功能（ARP）列表中。
  • The /unsafeuninstall command-line switch lets users attempt to "force" a bundle to uninstall, even if dependency checking would otherwise leave it installed.
    /unsafeuninstall 命令行开关允许用户尝试“强制”卸载捆绑包，即使依赖性检查会使其保持安装状态。

Deprecations and deletions​
弃用和删除

  • Features that were deprecated in WiX v3, including command-line switches deprecated in WiX v3.14, have been removed from WiX v4.
    在 WiX v3 中被弃用的功能，包括在 WiX v3.14 中被弃用的命令行开关，已从 WiX v4 中删除。
  • WixGamingExtension and WixLuxExtension have been removed in WiX v4.
    WixGamingExtension 和 WixLuxExtension 已在 WiX v4 中被移除。
  • WixDifxAppExtension is deprecated in Windows 10 and therefore has been deprecated in WiX v4 and will be removed in WiX v5.
    WixDifxAppExtension 在 Windows 10 中已被弃用，因此在 WiX v4 中也被弃用，并将在 WiX v5 中被移除。

    build -outputType 被忽略，来自 @robmen

    NetFxDotNetCompatibilityCheck 自定义操作严重碎片化，来自 @barnson

  • Substitution does not create ModuleSubstitution table when building Merge Module, from @rsdk-vag
    构建合并模块时，替换不会创建 ModuleSubstitution 表，来自 @rsdk-vag

  • RegisterFonts action is not added to the InstallExecuteSequence when fonts are being installed to the FontsFolder., from @kerrywicks
    注册字体操作在安装到字体文件夹时未添加到 InstallExecuteSequence 中，来自@kerrywicks

  • Migrated WiX v3 to v4, now getting error doing action Wix4ConfigureSmbUninstall_X64 when installing, from @barnson
    将 WiX v3 迁移到 v4，现在在安装时执行 Wix4ConfigureSmbUninstall_X64 操作时出现错误，来自@barnson

  • IWindowsInstallerDecompileContext.TreatProductAsModule is borked, from @barnson
    IWindowsInstallerDecompileContext.TreatProductAsModule 损坏，来自@barnson

  • wix msi decompile -x removes modularization GUIDs from object fields, from @barnson
    wix msi decompile -x 从对象字段中删除模块化 GUID，来自 @barnson

  • Merge modules don't extract during decompilation, from @barnson
    合并模块在反编译期间不会提取，来自 @barnson

All of the goodness in WiX v4.0.0 and v4.0.1 remains.

WiX v4 候选发布版 4 包含了与 RC3 相比显著减少的错误修复，这些错误是在候选发布版 3 期间报告的。还有一个小的新功能：

Release Candidate 3 released Friday, 24-February-2023
候选发布版 3 于 2023 年 2 月 24 日星期五发布

WiX v4 Release Candidate 3 contains fixes for bugs that were reported during Release Candidate 2. There are two minor new features:
WiX v4 候选发布版本 3 包含在候选发布版本 2 期间报告的错误修复。有两个新的小功能：

    WiX MSBuild 项目支持使用单个 ProjectReference 对其他项目进行多目标设置。您可以指定多个平台、配置、.NET 框架和运行时标识符。请参阅 @robmen 的 WIP 和拉取请求。
  • A new sys.BUILDARCHSHORT built-in preprocessor variable helps handle WiX v4's new architecture-specific custom actions. See issue 7227 and @barnson's PR.
    一个新的 sys.BUILDARCHSHORT 内置预处理器变量有助于处理 WiX v4 的新架构特定自定义操作。请参阅问题 7227 和 @barnson 的 PR。

WiX v4 候选版本 2 包含在候选版本 1 期间报告的错误修复。新增了两个功能：
