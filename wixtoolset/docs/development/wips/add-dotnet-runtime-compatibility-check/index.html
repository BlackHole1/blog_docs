Add DotNetCompatibilityCheck to the NetFx extension for the detection of .NET Core/.NET 5 runtimes
将 DotNetCompatibilityCheck 添加到 NetFx 扩展中以检测 .NET Core/.NET 5 运行时

User stories​ 用户故事

  • As a Setup developer I can specify a .NET runtime package in my bundle with built-in and robust runtime installation detection.
    作为一个安装开发人员，我可以在我的捆绑包中指定一个 .NET 运行时包，并内置 robust 运行时安装检测。

  • As a Setup developer I can run a check in my bundle that will determine if a specific version of the .NET runtime is installed such that the result can be used as an install condition for the runtime.
    作为一个安装开发人员，我可以在我的捆绑包中运行一个检查，以确定是否安装了特定版本的 .NET 运行时，从而可以将结果用作运行时的安装条件。

Proposal​ 提案

Right now, the NetFx extension supports .NET Core/.NET 5 runtime detection using a combination of path and registry checks. As discussed in dotnet/runtime#36479, this strategy is not sufficient for robust runtime installation detection.
目前，NetFx 扩展支持使用路径和注册表检查的组合来检测 .NET Core/.NET 5 运行时。正如在 dotnet/runtime#36479 中讨论的那样，这种策略不足以进行 robust 运行时安装检测。

An alternate solution is to use the the NETCoreCheck tool, built and published in the .NET repo here. NetCoreCheck is a CLI tool that can determine if a specific .NET runtime can be executed on the system. Using this tool in the NetFx extension will allow us to easily handle runtime dependency installation for .NET Core package definitions, including newer version detection #6257. The NetCoreCheck tool is already being used in Visual Studio by ClickOnce and Installer Projects for the same purpose.
另一种解决方案是使用在 .NET 仓库中构建和发布的 NETCoreCheck 工具。NetCoreCheck 是一个 CLI 工具，可以确定系统上是否可以执行特定的 .NET 运行时。在 NetFx 扩展中使用此工具将使我们能够轻松处理 .NET Core 包定义的运行时依赖项安装，包括较新版本检测 #6257。NetCoreCheck 工具已经在 Visual Studio 中被 ClickOnce 和安装项目用于相同的目的。

I'm proposing that we create a DotNetCompatibilityCheck element in the NetFx extension that will allow us to run the NETCoreCheck tool and use the result as a condition to install a .NET runtime package. The element will be implemented as an extension search as defined by this WIP. The NetCoreCheck tool will be included by the Burn engine as an embedded payload, which makes it available to the check through native code.
我建议我们在 NetFx 扩展中创建一个 DotNetCompatibilityCheck 元素，该元素将允许我们运行 NETCoreCheck 工具并使用结果作为安装 .NET 运行时包的条件。该元素将作为此 WIP 定义的扩展搜索实现。NetCoreCheck 工具将由 Burn 引擎作为嵌入式有效负载包含，从而使其通过本机代码可用于检查。

Using this approach will allow developers to avoid the question "is this software installed?", which can be misleading for .NET runtimes, in favor of "will my app run?". According to the .NET Core Releases and Support page, "Current" releases are supported for up to 3 months after the next major/minor release and "LTS" releases are supported for just 3 years. In addition, there are monthly servicing updates that are out of support as soon as the next one is released.
使用这种方法将使开发人员避免“此软件是否已安装？”这个问题，这对于 .NET 运行时来说可能会产生误导，而更倾向于“我的应用程序能运行吗？”。根据 .NET Core 发布和支持页面，“当前”版本在下一个主要/次要版本发布后最多支持 3 个月，“LTS”版本仅支持 3 年。此外，每月的服务更新在下一个版本发布后立即停止支持。

It is important for developers to keep their applications supported and secure. The easiest way to do that with such short periods of support for current releases is to utilize roll-forward policies. With this in mind, the question "will my app run on this PC?" becomes more important than "is this software installed?". If there is a newer version of the release available, developers will want their application to use that version.
对于开发人员来说，保持他们的应用程序受支持和安全是很重要的。由于当前版本的支持周期很短，最简单的方法是利用前滚策略。考虑到这一点，“我的应用程序能在这台电脑上运行吗？”这个问题比“这个软件安装了吗？”更重要。如果有更新版本的发布，开发人员会希望他们的应用程序使用该版本。

The NetCoreCheck tool (after some improvements) will allow applications that use roll-forward policies to accurately detect if the app will run on the target PC. Without this check, it's possible the installer would install a .NET release that the app doesn't even use when running. For example, if there’s a 3.1 app with roll-forward set to "LatestMajor" and it gets installed using this design on a machine with just 5.0, we would install the 3.1 runtime but then 5.0 would still be loaded when the app runs. This approach will also avoid the situation described in #6257 where an older patch version attempting to install over a newer one will fail.
NetCoreCheck 工具（经过一些改进）将允许使用前滚策略的应用程序准确检测该应用程序是否能在目标电脑上运行。如果没有这个检查，安装程序可能会安装一个应用程序运行时根本不使用的 .NET 版本。例如，如果有一个设置为“LatestMajor”的 3.1 应用程序，并且它在只有 5.0 的机器上使用这种设计安装，我们会安装 3.1 运行时，但在应用程序运行时仍会加载 5.0。这种方法还将避免 #6257 中描述的情况，即尝试安装旧版本 patch 覆盖新版本会失败。

The element will also be used for #6264, Luke's proposal for .NET Core/.NET 5 runtime installation checks with custom actions and launch conditions. To make this possible, the element must me made available to bundles as well as MSIs.
该元素还将用于 #6264，Luke 提出的 .NET Core/.NET 5 运行时安装检查的自定义操作和启动条件。为了实现这一点，该元素必须对捆绑包和 MSI 都可用。

I've designed a preliminary schema for DotNetCompatibilityCheck below:
我在下面设计了一个初步的 DotNetCompatibilityCheck 架构：

<xs:element name="DotNetCompatibilityCheck">
    <xs:annotation>
        <xs:documentation>Determines if an application that uses a specific version of the .NET runtime will run on the target machine.</xs:documentation>
        <xs:appinfo>
            <xse:parent namespace="http://wixtoolset.org/schemas/v4/wxs" ref="Bundle" />
            <xse:parent namespace="http://wixtoolset.org/schemas/v4/wxs" ref="Fragment" />
        </xs:appinfo>
    </xs:annotation>
    <xs:complexType>
        <xs:attribute name="Id" type="xs:string" use="required">
            <xs:annotation>
                <xs:documentation>
                    The identifier for this DotNetCompatibilityCheck.
                </xs:documentation>
            </xs:annotation>
        </xs:attribute>
        <xs:attribute name="RuntimeType" type="xs:string" use="required">
            <xs:annotation>
                <xs:documentation>
                    The type of .NET runtime being checked for.
                </xs:documentation>
            </xs:annotation>
            <xs:simpleType>
                <xs:restriction base="xs:NMTOKEN">
                    <xs:enumeration value="aspnet">
                        <xs:annotation>
                            <xs:documentation>
                                Attempt to check for an ASP.NET Core type runtime. The ASP.NET Core Runtime enables
                                you to run web/server applications.
                            </xs:documentation>
                        </xs:annotation>
                    </xs:enumeration>
                    <xs:enumeration value="desktop">
                        <xs:annotation>
                            <xs:documentation>
                                Attempt to check for a .NET Desktop type runtime. The .NET Desktop Runtime enables
                                you to run Windows desktop applications.
                            </xs:documentation>
                        </xs:annotation>
                    </xs:enumeration>
                    <xs:enumeration value="core">
                        <xs:annotation>
                            <xs:documentation>
                                Attempt to check for a .NET type runtime. The .NET Runtime contains just the components
                                needed to run a console app.
                            </xs:documentation>
                        </xs:annotation>
                    </xs:enumeration>
                </xs:restriction>
            </xs:simpleType>
        </xs:attribute>
        <xs:attribute name="Platform" type="xs:string" use="required">
            <xs:annotation>
                <xs:documentation>
                    Sets the platform for the .NET runtime being checked for.
                </xs:documentation>
            </xs:annotation>
            <xs:simpleType>
                <xs:restriction base="xs:NMTOKEN">
                    <xs:enumeration value="x86">
                        <xs:annotation>
                            <xs:documentation>
                                Attempt to check for the x86 version of a .NET runtime.
                            </xs:documentation>
                        </xs:annotation>
                    </xs:enumeration>
                    <xs:enumeration value="x64">
                        <xs:annotation>
                            <xs:documentation>
                                Attempt to check for the x64 version of a .NET runtime.
                            </xs:documentation>
                        </xs:annotation>
                    </xs:enumeration>
                    <xs:enumeration value="arm64">
                        <xs:annotation>
                            <xs:documentation>
                                Attempt to check for the ARM64 version of a .NET runtime.
                            </xs:documentation>
                        </xs:annotation>
                    </xs:enumeration>
                </xs:restriction>
            </xs:simpleType>
        </xs:attribute>
        <xs:attribute name="RollForward" type="xs:string" use="optional">
            <xs:annotation>
                <xs:documentation>
                    Sets the roll-forward policy that the application is using.
                </xs:documentation>
            </xs:annotation>
            <xs:simpleType>
                <xs:restriction base="xs:NMTOKEN">
                    <xs:enumeration value="latestpatch">
                        <xs:annotation>
                            <xs:documentation>
                                Roll forward to the highest patch version. This disables minor version roll forward.
                            </xs:documentation>
                        </xs:annotation>
                    </xs:enumeration>
                    <xs:enumeration value="minor">
                        <xs:annotation>
                            <xs:documentation>
                                Roll forward to the lowest higher minor version, if requested minor version is missing. If the requested minor version is present, then the latestpatch policy is used.
                            </xs:documentation>
                        </xs:annotation>
                    </xs:enumeration>
                    <xs:enumeration value="major">
                        <xs:annotation>
                            <xs:documentation>
                                Roll forward to lowest higher major version, and lowest minor version, if requested major version is missing. If the requested major version is present, then the minor policy is used.
                            </xs:documentation>
                        </xs:annotation>
                    </xs:enumeration>
                    <xs:enumeration value="latestminor">
                        <xs:annotation>
                            <xs:documentation>
                                Roll forward to highest minor version, even if requested minor version is present.
                            </xs:documentation>
                        </xs:annotation>
                    </xs:enumeration>
                    <xs:enumeration value="latestmajor">
                        <xs:annotation>
                            <xs:documentation>
                                Roll forward to highest major and highest minor version, even if requested major is present.
                            </xs:documentation>
                        </xs:annotation>
                    </xs:enumeration>
                    <xs:enumeration value="disable">
                        <xs:annotation>
                            <xs:documentation>
                                Do not roll forward. Only bind to specified version. This policy is not recommended for general use since it disable the ability to roll-forward to the latest patches. It is only recommended for testing.
                            </xs:documentation>
                        </xs:annotation>
                    </xs:enumeration>
                </xs:restriction>
            </xs:simpleType>
        </xs:attribute>
        <xs:attribute name="Version" type="xs:string" use="required">
            <xs:annotation>
                <xs:documentation>
                    The version of the .NET runtime being checked for (e.g. 3.1.10, 5.0.1).
                </xs:documentation>
            </xs:annotation>
        </xs:attribute>
        <xs:attribute name="Variable" type="xs:string" use="required">
            <xs:annotation>
                <xs:documentation>Name of the variable in which to place the result of the check.</xs:documentation>
            </xs:annotation>
        </xs:attribute>
        <xs:attribute name="Condition" type="xs:string">
            <xs:annotation>
                <xs:documentation>Condition for evaluating the check. If this evaluates to false, the check is not executed at all.</xs:documentation>
            </xs:annotation>
        </xs:attribute>
        <xs:attribute name="After" type="xs:string">
            <xs:annotation>
                <xs:documentation>Id of the element that this one should come after.</xs:documentation>
            </xs:annotation>
        </xs:attribute>
    </xs:complexType>
</xs:element>

A core component of this check is the NetCoreCheck exe which is a CLI tool that can be executed with the following arguments:
此检查的核心组件是 NetCoreCheck exe，这是一个可以使用以下参数执行的 CLI 工具：

  • Required framework name (Examples: "Microsoft.WindowsDesktop.App", "Microsoft.NETCore.App")
    所需的框架名称（示例：“Microsoft.WindowsDesktop.App”，“Microsoft.NETCore.App”）
  • Required major/minor/build framework version (Examples: "3.1.4", "5.0.0")
    所需的主要/次要/构建框架版本（示例：“3.1.4”，“5.0.0”）
  • Optional roll-forward policy (Example: "LatestPatch", "Minor")
    可选的前滚策略（示例：“LatestPatch”，“Minor”）
    • Support for this argument is under development and tracked by dotnet/deployment-tools/issues/82.
      对此参数的支持正在开发中，并由 dotnet/deployment-tools/issues/82 跟踪。

And returns the following codes:
并返回以下代码：

  • Success: 0, Means the required runtime is installed on the machine
    成功：0，表示所需的运行时已安装在机器上。
  • Failure: 失败：
    • 1 - Failed to load HostFxr, meaning no runtime is installed
      1 - 加载 HostFxr 失败，意味着没有安装运行时
    • 2 - Failed to initialize HostFxr for the config, meaning the required runtime isn't installed
      2 - 初始化 HostFxr 失败，意味着未安装所需的运行时
    • 3 - Invalid Args 3 - 无效参数
    • Other - Unexpected failures
      其他 - 意外故障

Examples: 示例:

  • NetCoreCheck.exe Microsoft.WindowsDesktop.App 3.1.4
  • NetCoreCheck.exe Microsoft.NETCore.App 5.0.0

It should be noted that there are two versions of the NetCoreCheck exe (Win-x64/Win-x86). Support for ARM64 is under development and tracked by dotnet/deployment-tools/issues/81;
请注意，NetCoreCheck exe 有两个版本（Win-x64/Win-x86）。对 ARM64 的支持正在开发中，并由 dotnet/deployment-tools/issues/81 跟踪。

The check can be used by Setup developers to detect .NET runtime installations as an installation condition for a .NET runtime package:
该检查可供安装程序开发人员用作 .NET 运行时包的安装条件来检测 .NET 运行时安装：

<?xml version="1.0"?>
<Wix xmlns="http://schemas.microsoft.com/wix/2006/wi"
     xmlns:netfx="http://schemas.microsoft.com/wix/NetFxExtension">
  <Fragment>
    <netfx:DotNetCompatibilityCheck Id="Runtime"
        RuntimeType="Desktop"
        Platform="x86"
        Version="3.1.10"
        Variable="DotNetCompatibilityCheckResult" />

    <PackageGroup Id="MyPackage">
      <ExePackage 
        SourceFile="[sources]\packages\shared\windowsdesktop-runtime-3.1.10-win-x86.exe"
        DownloadURL="https://download.visualstudio.microsoft.com/download/pr/865d0be5-16e2-4b3d-a990-f4c45acd280c/ec867d0a4793c0b180bae85bc3a4f329/windowsdesktop-runtime-3.1.10-win-x86.exe"
        InstallCommand="/q /ACTION=Install"
        RepairCommand="/q ACTION=Repair /hideconsole"
        UninstallCommand="/q ACTION=Uninstall /hideconsole"
        InstallCondition="x86 = 1 AND OSVersion >= v5.0.5121.0 AND DotNetCompatibilityCheckResult = 0" />
    </PackageGroup>
  </Fragment>
</Wix>

Considerations​ 考虑事项

DotNetCompatibilityCheck Implementation​
DotNet 兼容性检查实现

There are a few different ways that we can implement .NET Core/.NET 5 runtime detection:
我们可以通过几种不同的方式实现 .NET Core/.NET 5 运行时检测：

  1. Run the NetCoreCheck exe to determine if a suitable runtime is installed by loading the .NET runtime. The NetCoreCheck tool will:
    运行 NetCoreCheck 可执行文件，通过加载 .NET 运行时来确定是否安装了合适的运行时。NetCoreCheck 工具将：
    • Attempt to locate hostfxr.dll using get_hostfxr_path.
      尝试使用 get_hostfxr_path 定位 hostfxr.dll。
    • Generate a temporary runtimeconfig.json file based on the runtime.
      根据运行时生成一个临时的 runtimeconfig.json 文件。
    • Load library hostfxr.dll and call hostfxr_initialize_for_runtime_config using the generated runtimeconfig.json file, success here means the necessary runtime is installed.
      加载library hostfxr.dll 并使用生成的 runtimeconfig.json 文件调用 hostfxr_initialize_for_runtime_config，这里成功意味着已安装必要的运行时。
    • Call hostfxr_close and unload hostfxr library.
      调用 hostfxr_close 并卸载 hostfxr library。
  2. Parse the output of dotnet.exe --list-runtimes to determine which runtimes are installed (suggested by rseanhall in #6264).
    解析 dotnet.exe --list-runtimes 的输出以确定安装了哪些运行时（由 rseanhall 在 #6264 中建议）。
    • More research will be needed to determine whether to try to locate the different architectures of dotnet.exe or create our own exes to run it through hostfxr_main.
      需要更多研究来确定是尝试定位 dotnet.exe 的不同架构还是创建我们自己的可执行文件通过 hostfxr_main 运行它。
  3. Use the solution outlined in dotnet/sdk/15021 which involves using host fxr APIs along with installed bundle directory checks.
    使用 dotnet/sdk/15021 中概述的解决方案，该解决方案涉及使用 host fxr API 以及已安装的捆绑目录检查。
    • This is how dotnet.exe currently works.
      这就是 dotnet.exe 目前的工作方式。

Approach 1 is what I have proposed in the previous section. rseanhall mentioned that it could be more efficient to write code that could be shared with #6264. One question I'm not sure about: can the NetCoreCheck-in-binary-table approach work for both #6257: Burn chain and #6264: Custom actions/launch conditions?
方法 1 是我在上一节中提出的。rseanhall 提到编写可以与#6264 共享的代码可能更高效。我不确定的一个问题是：NetCoreCheck-in-binary-table 方法是否可以同时适用于#6257: Burn chain 和#6264: Custom actions/launch conditions？

In dotnet/runtime/36479, vitek-karas addresses approach 2 (parsing the output of --list-runtimes):
在 dotnet/runtime/36479 中，vitek-karas 讨论了方法 2（解析--list-runtimes 的输出）：

"I think the proposal (for this approach) is to add wrapper tool around the dotnet --list-runtimes that does the text processing and provides a return value, so that individual installers don't have to worry about it."
我认为这个方法的提议是添加一个包装工具，围绕 dotnet --list-runtimes 进行文本处理并提供返回值，这样各个安装程序就不必担心这个问题了。

My understanding of the ask here is to answer a question like "Will application requiring framework XYZ run on this machine?". Answering that question from just a list of installed frameworks is actually non-trivial. The entire machinery of framework resolution, version conflict resolution and roll-forward is on top of the flat list. That's a lot of logic. So I don't think that parsing the dotnet.exe output is the right approach here.
我对这里要求的理解是回答一个类似“需要框架 XYZ 的应用程序能在这台机器上运行吗？”的问题。仅从已安装框架的列表中回答这个问题实际上并不简单。框架解析、版本冲突解析和前滚的整个机制都在平面列表之上。这涉及很多逻辑。所以我认为解析 dotnet.exe 输出不是这里的正确方法。

I'm leaning towards the NetCoreCheck approach because it already exists, has been validated, and answers the valuable question of "will my app run?" instead of "is this software installed?".
我倾向于使用 NetCoreCheck 方法，因为它已经存在，已经过验证，并且回答了“我的应用程序能运行吗？”这个有价值的问题，而不是“这个软件是否已安装？”。

Regarding approach 3, the hostfxr API in question is not currently available. Should it become available, we can easily add the functionality it provides to the NetCoreCheck tool.
关于方法 3，所讨论的 hostfxr API 目前不可用。如果它变得可用，我们可以轻松地将其提供的功能添加到 NetCoreCheck 工具中。
