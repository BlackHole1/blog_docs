WiX Improvement Proposals
WiX 改进提案

User stories​ 用户故事

  • As a WiX developer I can document changes I plan to make to the WiX Toolset features or processes such that other developers can discuss the proposal.
    作为一名 WiX 开发人员，我可以记录我计划对 WiX 工具集功能或流程进行的更改，以便其他开发人员可以讨论该提案。

  • As a WiX developer I can summarize all of the discussion about an improvement such that future developers and users can find the results in one document.
    作为一名 WiX 开发人员，我可以总结所有关于改进的讨论，以便未来的开发人员和用户可以在一份文件中找到结果。

Proposal​ 提案

A WiX Improvement Proposal, or WIP, is a document that defines a process or feature improvement in the WiX Toolset. Not every change to the WiX Toolset must have a WIP. Bug fixes, for example, are often sufficiently defined in the issue tracker. For everything else the WIP process is intended to be a lightweight process that provides value for developers today and tomorrow by recording the data that went into a decision and providing "institutional memory" for everyone working on the WiX Toolset.
WiX 改进提案（WIP）是定义 WiX 工具集中流程或功能改进的文档。并非对 WiX 工具集的每一次更改都必须有 WIP。例如，错误修复通常在问题跟踪器中有足够的定义。对于其他所有内容，WIP 流程旨在成为一个轻量级流程，通过记录决策所依据的数据并为所有从事 WiX 工具集工作的人员提供“机构记忆”，为今天和明天的开发人员提供价值。

Proposing a change with a WIP​
提出一个 WIP 的变更

The following list describes one approach for improving WiX using a WIP:
以下列表描述了一种使用 WIP 改进 WiX 的方法：

  • Have an idea for improving WiX.
    有一个改进 WiX 的想法。

  • Break it down, if necessary, to discrete ideas. We don't want to lose the big picture but breaking down big ideas helps ensure you've thought through all the steps and helps others better understand your proposal.
    如果有必要，将其分解为离散的想法。我们不想失去大局，但分解大想法有助于确保你已经考虑了所有步骤，并帮助其他人更好地理解你的提议。

  • Open a feature request on the WiX Toolset issue tracker. Provide enough detail about what the improvement does and how it makes WiX better; that allows the triage team and those attending our fortnightly triage meetings to offer feedback and, if all goes well, schedule the feature in an appropriate release.
    在 WiX 工具集问题跟踪器上打开一个功能请求。提供足够的细节说明改进的内容以及如何使 WiX 更好；这使得分类团队和参加我们每两周一次分类会议的人能够提供反馈，如果一切顺利，将该功能安排在适当的版本中。

  • Draft your WIP. Get the ideas "down on paper." Send a pull request to get feedback on the draft and updated in the WiX Git repository.
    起草你的 WIP。把想法“写在纸上”。发送一个拉取请求以获取对草稿的反馈，并在 WiX Git 存储库中更新。

  • Get public feedback. Start with a post on the WiX Development category in the WiX discussion area on GitHub.
    获取公众反馈。从 GitHub 上的 WiX 讨论区的 WiX 开发类别开始发帖。

  • Capture the feedback. Use the feedback to improve your WIP. Add useful suggestions. Address weaknesses. List alternatives. Consider what it means if you're getting lots of negative feedback. (But don't let the haters get to you.)
    收集反馈。利用反馈改进你的 WIP。添加有用的建议。解决弱点。列出替代方案。如果你收到大量负面反馈，考虑这意味着什么。（但不要让仇恨者影响你。）

  • Update your WIP. Complete all the sections of the WIP. Send pull requests to get feedback from the WiX leads and others. When you're done, send the feature request back for triage to get final approval.
    更新你的 WIP。完成 WIP 的所有部分。发送拉取请求以获得 WiX 负责人和其他人的反馈。当你完成后，将功能请求发送回分类以获得最终批准。

That's not a hard set of rules; nor do you have to do them in that order serially. You could finish coding your change before writing one word in a WIP, for example. The risk in doing so is that you miss out on feedback when you could incorporate it at relatively low cost -- that is, relatively little rework. It's better to overcommunicate and overdiscuss.
这不是一套严格的规则；你也不必按顺序依次完成它们。例如，你可以在编写 WIP 之前完成代码更改。这样做的风险在于你可能错过了可以以相对较低的成本（即相对较少的返工）进行反馈的机会。最好是过度沟通和讨论。

Creating a WiX Improvement Proposal​
创建 WiX 改进提案

WIPs live in the comments of the bug or feature request in the issue tracker. GitHub lets you edit your own comments so you can update the WIPs as often as necessary (and previous comment versions are available for your viewing pleasure).
WIP 存在于问题跟踪器中的错误或功能请求的评论中。GitHub 允许你编辑自己的评论，因此你可以根据需要频繁更新 WIP（并且可以查看以前的评论版本）。

WIP User Stories​ 进行中的用户故事

The WIP starts with user stories. Follow the standard user story format "As a role, I can accomplish task thanks to proposal such that benefit from proposal." There are a couple of standard roles in WiX:
进行中的用户故事从用户故事开始。遵循标准的用户故事格式“作为一个角色，我可以完成任务，因为提案使我受益。” WiX 中有几个标准角色：

  • WiX developer - someone like you that works on the WiX Toolset itself to improve it.
    WiX 开发者 - 像你这样的人，致力于改进 WiX 工具集。

  • Setup developer - a user of the WiX Toolset that builds installation packages.
    设置开发者 - WiX 工具集的用户，用于构建安装包。

Ideally, there is more than one user story.
理想情况下，有多个用户故事。

WIP Proposal​ 工作中的提案

The proposal section of the WIP is a fairly freeform description of the feature or process improvement. Provide the motivation for the proposal. Describe how the proposal works. List examples, particularly source code, that demonstrates how the improvement will be used. Basically, provide the bulk of the detail about the WIP here.
WIP 的提案部分是对功能或流程改进的相当自由的描述。提供提案的动机。描述提案的工作原理。列出示例，特别是源代码，展示改进将如何使用。基本上，在这里提供关于 WIP 的大部分细节。

WIP Considerations​ WIP 考虑事项

As issues with the proposal come to mind or are raised by reviewers, capture them in the considerations section of the WIP. As appropriate, explain the trade offs of the proposed design and why a particular decision was made. This section may provide the most value for developers and users that follow later so do not be afraid to list everything that comes to mind. Rare is it that a proposal doesn't have some negatives to consider.
当提案出现问题或被审阅者提出时，请在 WIP 的考虑事项部分记录它们。根据需要，解释所提设计的权衡取舍以及为什么做出特定决定。这个部分可能为后续的开发人员和用户提供最大的价值，所以不要害怕列出所有想到的内容。很少有提案没有需要考虑的负面因素。

WIP Workflow​ WIP 工作流程

Like the code of the WiX Toolset, WIPs are designed to be living documents. So do not be afraid to "check in early and often." Update the issue comments that contain the WIP content whenever you feel the update provides enough value.
像 WiX 工具集的代码一样，WIP 旨在成为活文档。所以不要害怕“尽早并经常提交”。每当您觉得更新提供了足够的价值时，请更新包含 WIP 内容的问题评论。

WIPs can be discussed anywhere but everyone is encouraged to contribute their ideas to the proposal by adding comments to the issue or working with the author to update the document.
WIP 可以在任何地方讨论，但鼓励每个人通过在问题中添加评论或与作者合作更新文档来为提案贡献他们的想法。

Considerations​ 考虑事项

The WIP process is designed to add value with as little overhead as possible. Of course, it takes effort to write down how a feature should work. But often the process of writing down how to solve a problem, uncovers issues that were not initially considered. Capturing some of the alternatives or objections that came up during development is of significant value, especially for future developers. So, while it takes a bit more time up front, we think the WIPs will add a lot of value over the long term.
WIP 流程旨在以尽可能少的开销增加价值。当然，写下一个功能应该如何工作需要付出努力。但通常写下如何解决问题的过程会发现最初没有考虑到的问题。记录在开发过程中出现的一些替代方案或反对意见具有重要价值，尤其是对未来的开发人员。因此，虽然前期需要花费更多时间，但我们认为 WIP 将在长期内增加很多价值。

Finally, this is not a college writing assignment. Use as many -- or as few -- words as necessary to capture the idea. No need to write a book. Also, feel free to enlist others to help you write the document. Start with a rough draft and have someone help flesh out the missing details.
最后，这不是大学写作作业。用尽可能多或尽可能少的词语来表达想法。无需写一本书。此外，随时可以请其他人帮你写文档。从草稿开始，让别人帮你补充缺失的细节。
