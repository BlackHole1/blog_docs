Code Style 代码风格

This document defines the WiX code style that is adapted from original source code developed at Microsoft. This is a living document and you will find older source in the repository that do not adhere to all of the guidelines in this document. When relevant to do so please fix the code to match this document.
本文档定义了从微软开发的原始源代码中改编的 WiX 代码风格。这是一个动态文档，您会在存储库中找到不符合本文档中所有指南的旧源代码。在相关情况下，请修正代码以符合本文档。

  1. Do not use tabs - only spaces.
    不要使用制表符 - 只使用空格。

  2. The left indent should be four spaces.
    左缩进应为四个空格。

  3. Insignificant whitespace (whitespace to the right of text) should be reduced to one space in all cases except maybe comments.
    无关紧要的空白（文本右侧的空白）应减少到一个空格，除非是注释。

    // Good
    int a = 0;
    DWORD dw = 0; // this exists for some good reason

    // Bad
    int   a  = 0;
    DWORD dw = 0;   // this exists for some good reason

  4. Control flow operators get one space after them while functions do not.
    控制流操作符后面应有一个空格，而函数则不需要。

    // Good
    if (x)
    {
        func(x, y, z);
    }

    do
    {
        // do something to change fGo eventually
    } while (fGo);

    // Bad
    if( x )
    {
        func ( x, y, z );
    }

  5. Braces are always left justified.
    大括号总是左对齐。

    // Good
    HRESULT FunctionName(int a)
    {
        int b = 0;
        int c = 0;

        if (0 == a)
        {
            b = 1;
            c = 2;
        }
    }

    // Bad
    HRESULT FunctionName(int a) {
        int a = 0;
        int b = 0;

        if (0 == a)
            {
            b = 1;
            c = 2;
            }
    }

  6. Braces are always used, even when optional.
    始终使用大括号，即使在可选时也是如此。

    // Good
    if (fBar)
    {
        a = 1;
    }

    // Bad
    if (fBar)
        a = 1;

  7. Function definitions should look something like these examples.
    函数定义应类似于这些示例。

    // Good
    void FunctionName();

    extern "C" HRESULT __stdcall FunctionName(
        __in LPCWSTR wzFoo,
        __out BOOL *pfBar
        );

    // Bad
    void FunctionName(
        );

    extern "C"
    HRESULT
    __stdcall
    FunctionName(
        __in LPCWSTR szFoo,
        __out BOOL *pfBar
        );

  8. Use prefix addition and subtraction except where postfix addition or subtraction is absolutely required. Avoid postfix in general to avoid side effects.
    使用前缀加法和减法，除非绝对需要后缀加法或减法。一般避免使用后缀以避免副作用。

    // Good
    for (int i = 0; i < 10; ++i)
    {
    }

    // Bad
    for (int i = 0; i < 10; i++)
    {
    }

  9. Infinite loops should be rare but when necessary use an empty for loop and remember the braces.
    无限循环应该很少见，但在必要时使用空的 for 循环并记住大括号。

    // Example infinite loop required for some reason
    for (;;)
    {
        // eventually break
    }

  10. SAL annotation is required on all function parameters.
    所有函数参数都需要 SAL 注释。

  11. Functions should have only one of the following return types: void, BOOL, or HRESULT. void and BOOL functions can never fail. If they could fail, change the return type to an HRESULT. You will find lots of wrapper functions in dutil.lib that all return HRESULTs. The only exception to this rule is that "uninitialize"-type functions should return void. Work hard to remove any failure cases in the uninitialize route and, if unsuccessful, carefully consider what implications a failed uninitialize route has. Consider ::RevertToSelf() as an example.
    函数应该只有以下返回类型之一： void 、 BOOL 或 HRESULT 。 void 和 BOOL 函数永远不会失败。如果它们可能失败，请将返回类型更改为 HRESULT 。你会在 dutil.lib 中找到很多返回 HRESULT 的包装函数。此规则的唯一例外是“反初始化”类型的函数应返回 void 。努力消除反初始化路径中的任何失败情况，如果不成功，请仔细考虑失败的反初始化路径的影响。以 ::RevertToSelf() 为例。

  12. The C++ type bool is never used. Use BOOL instead.
    C++类型 bool 从未使用过。请改用 BOOL 。

  13. Functions that are from a public SDK should be prefixed with the global namespace scope operator ::.
    来自公共 SDK 的函数应以全局namespace作用域操作符 :: 为前缀。

  14. Use the countof() macro (defined in dutil.h) to get the number of elements in an array. Consider this a sibling of the sizeof() language operator.
    使用 countof() macro（在 dutil.h 中定义）来获取数组中的元素数量。将其视为 sizeof() 语言操作符的同类。

  15. There should only be one exit from a function. An exception to this is error checking that can circumvent significant variable initialization by existing immediately at the top of the function.
    一个函数应该只有一个出口。一个例外是错误检查，可以通过在函数顶部立即退出来避免大量变量初始化。

  16. Local variables should be initialized at the top of the function. If the local variable is not initialized to zero add a comment why.
    局部变量应在函数顶部初始化。如果局部变量未初始化为零，请添加注释说明原因。

  17. The l-value in conditional tests should be the constant value whenever possible. To some people this makes the logic look "backward" but prevents accidental assignment in native code. After a while, other ways start looking wrong.
    在条件测试中，左值应尽可能是常量值。对某些人来说，这使得逻辑看起来“反向”，但可以防止在本地代码中意外赋值。过一段时间后，其他方式开始显得不对劲。

    // Example
    if (1 == i)
    {
        ++i;
    }

  18. There are two "secret handshakes" in the WiX code style. First, the ExitOnXxx(), ExitWithXxx(), and ExitFunction() macros are used to simplify code flow. Never use an explicit goto. Second, the ReleaseXxx() and ReleaseNullXxx() macros are used extensively to clean up dynamically allocated memory when the function exits.
    在 WiX 代码风格中有两个“秘密握手”。首先， ExitOnXxx() 、 ExitWithXxx() 和 ExitFunction() 宏用于简化代码流程。绝不要使用显式的 goto 。其次， ReleaseXxx() 和 ReleaseNullXxx() 宏被广泛用于在函数退出时清理动态分配的内存。

    // Example
    HRESULT FunctionName()
    {
        HRESULT hr = S_OK;
        IUnknown *pObject = NULL;
        LPVOID pv = NULL;
        WCHAR wzPAth[MAX_PATH] = { };

        pv = MemAlloc(50, TRUE);
        ExitOnNull(pv, hr, E_OUTOFMEMORY, "Failed to allocate pointer to nothing.");

        hr = ::CoCreateInstance(IID_IUnknown, &pObject);
        ExitOnFailure(hr, "Failed to get fake COM object.");

        if (!::GetTempPath(wzPath, countof(wzPath)))
        {
            ExitWithLastError(hr, "Failed to get TEMP path.");
        }

    LExit: // this is always present when control flow is used
        ReleaseObject(pObject);
        ReleaseMem(pv);

        return hr;
    }

  19. The WiX support libraries make extensive use of dynamic strings. Use these string extensively instead of constant buffers or other mechanisms. You should use Hungarian notation for dynamic strings to change from "pwz" (pointer to a null-terminated string) to "scz" (for counted null-terminated string). This will make it easier to read "pointer to dynamically-allocated string" ("pscz" instead of "ppwz").
    WiX 支持库广泛使用动态字符串。请广泛使用这些字符串，而不是常量缓冲区或其他机制。你应该使用匈牙利命名法来将动态字符串从“pwz”（指向以空字符结尾的字符串的指针）改为“scz”（计数的以空字符结尾的字符串）。这将使阅读“指向动态分配字符串的指针”（“pscz”而不是“ppwz”）更容易。

  20. Use Hungarian notation for native code but don't go overboard with it. Sometimes just pProperName is just fine for an object that you are pointing at.
    对于本地代码使用匈牙利命名法，但不要过度。有时仅仅 pProperName 对于你指向的对象来说就足够了。

    BOOL f = FALSE;
    int i = 0;
    long l = 0;
    DWORD dw = 0;
    LPVOID pv = NULL;
    FUNCTION_PTR pfn = NULL;
    HRESULT hr = S_OK;

  21. Use the COM IUnknown pattern for lifetime management of objects. AddRef() and Release() are not perfect but they are better than any other memory tracking operations we've found have in native code.
    使用 COM IUnknown 模式进行对象的生命周期管理。 AddRef() 和 Release() 并不完美，但它们比我们在本地代码中发现的任何其他内存跟踪操作都要好。

  22. WiX code style for managed code follows StyleCop definitions. Most of the above native code has actually morphed gently to match StyleCop suggested design (i.e. no tabs, four spaces).
    WiX 管理代码的代码风格遵循 StyleCop 定义。大多数上述本地代码实际上已经轻微变形以匹配 StyleCop 建议的设计（即无制表符，四个空格）。
