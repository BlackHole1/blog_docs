MajorUpgrade Element 主要升级元素

Description 描述
Simplifies authoring for major upgrades, including support for preventing downgrades.
简化了主要升级的编写，包括支持防止降级。

The parent Product element must have valid UpgradeCode and Version attributes.
父产品元素必须具有有效的升级代码和版本属性。

When the FindRelatedProducts action detects a related product installed on the system, it appends the product code to the property named WIX_UPGRADE_DETECTED. After the FindRelatedProducts action is run, the value of the WIX_UPGRADE_DETECTED property is a list of product codes, separated by semicolons (;), detected on the system.
当 FindRelatedProducts 操作检测到系统上安装了相关产品时，它会将产品代码附加到名为 WIX_UPGRADE_DETECTED 的属性中。运行 FindRelatedProducts 操作后，WIX_UPGRADE_DETECTED 属性的值是系统上检测到的产品代码列表，代码之间用分号 (;) 分隔。

Attributes 属性
NameTypeDescriptionRequired
AllowDowngradesYesNoType When set to no (the default), products with lower version numbers are blocked from installing when a product with a higher version is installed; the DowngradeErrorMessage attribute must also be specified.
当设置为 no（默认值）时，如果安装了版本号较高的产品，则阻止安装版本号较低的产品；还必须指定 DowngradeErrorMessage 属性。

When set to yes, any version can be installed over any other version.
如果设置为是，任何版本都可以安装在任何其他版本之上。		 
AllowSameVersionUpgradesYesNoType When set to no (the default), installing a product with the same version and upgrade code (but different product code) is allowed and treated by MSI as two products. When set to yes, WiX sets the msidbUpgradeAttributesVersionMaxInclusive attribute, which tells MSI to treat a product with the same version as a major upgrade.
如果设置为否（默认），允许安装具有相同版本和升级代码（但不同产品代码）的产品，并且 MSI 将其视为两个产品。如果设置为是，WiX 会设置 msidbUpgradeAttributesVersionMaxInclusive 属性，这告诉 MSI 将具有相同版本的产品视为重大升级。

This is useful when two product versions differ only in the fourth version field. MSI specifically ignores that field when comparing product versions, so two products that differ only in the fourth version field are the same product and need this attribute set to yes to be detected.
当两个产品版本仅在第四版本字段中有所不同时，这很有用。MSI 在比较产品版本时会特别忽略该字段，因此仅在第四版本字段中有所不同的两个产品是相同的产品，需要将此属性设置为是才能被检测到。

Note that because MSI ignores the fourth product version field, setting this attribute to yes also allows downgrades when the first three product version fields are identical. For example, product version 1.0.0.1 will "upgrade" 1.0.0.2998 because they're seen as the same version (1.0.0). That could reintroduce serious bugs so the safest choice is to change the first three version fields and omit this attribute to get the default of no.
请注意，由于 MSI 忽略了第四个产品版本字段，将此属性设置为是也允许在前三个产品版本字段相同的情况下降级。例如，产品版本 1.0.0.1 将“升级”1.0.0.2998，因为它们被视为相同版本（1.0.0）。这可能会重新引入严重的错误，因此最安全的选择是更改前三个版本字段并省略此属性以获得默认的否。

This attribute cannot be "yes" when AllowDowngrades is also "yes" -- AllowDowngrades already allows two products with the same version number to upgrade each other.
当 AllowDowngrades 也为“是”时，此属性不能为“是”——AllowDowngrades 已经允许两个具有相同版本号的产品相互升级。		 
DisallowYesNoType When set to yes, products with higer version numbers are blocked from installing when a product with a lower version is installed; the UpgradeErrorMessage attribute must also be specified.
当设置为是时，当安装了较低版本的产品时，较高版本号的产品将被阻止安装；还必须指定 UpgradeErrorMessage 属性。

When set to no (the default), any version can be installed over any lower version.
设置为“否”（默认）时，任何版本都可以安装在任何较低版本之上。		 
DisallowUpgradeErrorMessageString The message displayed if users try to install a product with a higer version number when a product with a lower version is installed. Used only when Disallow is yes.
当用户尝试在安装了较低版本的产品时安装具有更高版本号的产品时显示的消息。仅在“Disallow”为“是”时使用。		 
DowngradeErrorMessageString The message displayed if users try to install a product with a lower version number when a product with a higher version is installed. Used only when AllowDowngrades is no (the default).
当用户尝试在安装了较高版本的产品时安装具有较低版本号的产品时显示的消息。仅在“AllowDowngrades”为“否”（默认）时使用。		 
IgnoreRemoveFailureYesNoType When set to yes, failures removing the installed product during the upgrade will be ignored.
设置为“是”时，将忽略在升级过程中删除已安装产品的失败。

When set to no (the default), failures removing the installed product during the upgrade will be considered a failure and, depending on the scheduling, roll back the upgrade.
设置为“否”（默认值）时，在升级过程中删除已安装产品的失败将被视为失败，并且根据调度情况，可能会回滚升级。		 
MigrateFeaturesYesNoType When set to yes (the default), the MigrateFeatureStates standard action will set the feature states of the upgrade product to those of the installed product.
设置为“是”（默认值）时，MigrateFeatureStates 标准操作将把升级产品的功能状态设置为已安装产品的功能状态。

When set to no, the installed features have no effect on the upgrade installation.
设置为“否”时，已安装的功能对升级安装没有影响。		 
RemoveFeaturesString A formatted string that contains the list of features to remove from the installed product. The default is to remove all features. Note that if you use formatted property values that evaluate to an empty string, no features will be removed; only omitting this attribute defaults to removing all features.
一个格式化的字符串，包含要从已安装产品中移除的功能列表。默认情况下会移除所有功能。请注意，如果使用评估为空字符串的格式化属性值，则不会移除任何功能；仅省略此属性默认移除所有功能。		 
ScheduleEnumeration Determines the scheduling of the RemoveExistingProducts standard action, which is when the installed product is removed. The default is "afterInstallValidate" which removes the installed product entirely before installing the upgrade product. It's slowest but gives the most flexibility in changing components and features in the upgrade product.
确定 RemoveExistingProducts 标准操作的调度时间，即何时移除已安装的产品。默认值是“afterInstallValidate”，即在安装升级产品之前完全移除已安装的产品。虽然速度最慢，但在更改升级产品中的组件和功能时提供了最大的灵活性。

For more information, see RemoveExistingProducts. This attribute's value must be one of the following:
有关更多信息，请参见 RemoveExistingProducts。此属性的值必须是以下之一：
afterInstallValidate
(Default) Schedules RemoveExistingProducts after the InstallValidate standard action. This scheduling removes the installed product entirely before installing the upgrade product. It's slowest but gives the most flexibility in changing components and features in the upgrade product. Note that if the installation of the upgrade product fails, the machine will have neither version installed.
（默认）在 InstallValidate 标准操作之后安排 RemoveExistingProducts。此调度在安装升级产品之前完全删除已安装的产品。它是最慢的，但在更改升级产品中的组件和功能方面提供了最大的灵活性。请注意，如果升级产品的安装失败，机器将不会安装任何版本。
afterInstallInitialize 安装后初始化
Schedules RemoveExistingProducts after the InstallInitialize standard action. This is similar to the afterInstallValidate scheduling, but if the installation of the upgrade product fails, Windows Installer also rolls back the removal of the installed product -- in other words, reinstalls it.
在 InstallInitialize 标准操作之后安排 RemoveExistingProducts。这类似于 afterInstallValidate 调度，但如果升级产品的安装失败，Windows Installer 还会回滚已安装产品的删除——换句话说，重新安装它。
afterInstallExecute 安装后执行
Schedules RemoveExistingProducts between the InstallExecute and InstallFinalize standard actions. This scheduling installs the upgrade product "on top of" the installed product then lets RemoveExistingProducts uninstall any components that don't also exist in the upgrade product. Note that this scheduling requires strict adherence to the component rules because it relies on component reference counts to be accurate during installation of the upgrade product and removal of the installed product. For more information, see Bob Arnson's blog post "Paying for Upgrades" for details. If installation of the upgrade product fails, Windows Installer also rolls back the removal of the installed product -- in other words, reinstalls it.
在 InstallExecute 和 InstallFinalize 标准操作之间安排 RemoveExistingProducts。此调度将在已安装产品之上安装升级产品，然后让 RemoveExistingProducts 卸载升级产品中不存在的任何组件。请注意，此调度要求严格遵守组件规则，因为它依赖于组件引用计数在升级产品安装和已安装产品移除期间的准确性。有关更多信息，请参阅 Bob Arnson 的博客文章“Paying for Upgrades”了解详细信息。如果升级产品的安装失败，Windows Installer 还会回滚已安装产品的移除——换句话说，就是重新安装它。
afterInstallExecuteAgain 在 InstallExecuteAgain 之后
Schedules RemoveExistingProducts between the InstallExecuteAgain and InstallFinalize standard actions. This is identical to the afterInstallExecute scheduling but after the InstallExecuteAgain standard action instead of InstallExecute.
在 InstallExecuteAgain 和 InstallFinalize 标准操作之间安排 RemoveExistingProducts。这与 InstallExecute 之后的调度相同，但在 InstallExecuteAgain 标准操作之后而不是 InstallExecute。
afterInstallFinalize 安装后完成
Schedules RemoveExistingProducts after the InstallFinalize standard action. This is similar to the afterInstallExecute and afterInstallExecuteAgain schedulings but takes place outside the installation transaction so if installation of the upgrade product fails, Windows Installer does not roll back the removal of the installed product, so the machine will have both versions installed.
在 InstallFinalize 标准操作之后安排 RemoveExistingProducts。这类似于 afterInstallExecute 和 afterInstallExecuteAgain 的调度，但发生在安装事务之外，因此如果升级产品的安装失败，Windows Installer 不会回滚已安装产品的移除，因此机器将安装两个版本。
