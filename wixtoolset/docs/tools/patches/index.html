Windows Installer patches
Windows 安装程序补丁

Windows Installer patching allows you to update an installed product without a full upgrade. Patches contain the differences between an older version of a package (called the target) and the newer version (the update). Because they contain only the differences, patches are typically smaller than the full update package and typically install more quickly.
Windows Installer 补丁允许您在不进行完整升级的情况下更新已安装的产品。补丁包含包的旧版本（称为目标）和新版本（更新）之间的差异。因为它们只包含差异，所以补丁通常比完整的更新包小，并且通常安装得更快。

That said, patching is an advanced topic and requires strict adherence to all the rules that come with Windows Installer. Using major upgrades with "early" scheduling, such as using the default Schedule attribute of the MajorUpgrade element is far simpler. As Bob Arnson discussed way back in 2008:
话虽如此，补丁是一个高级主题，需要严格遵守 Windows Installer 的所有规则。使用带有“早期”调度的主要升级，例如使用 MajorUpgrade 元素的默认 Schedule 属性要简单得多。正如 Bob Arnson 在 2008 年早些时候讨论的那样：

If you don’t absolutely need to ship patches, you can avoid the costs of minor upgrades by simply using major upgrades. You can remove files without worrying about component-rule violations if you use an “early” scheduling of the RemoveExistingProducts standard action – before or immediately after the InstallInitialize action.
如果您不绝对需要发布补丁，您可以通过简单地使用主要升级来避免小升级的成本。如果您使用 RemoveExistingProducts 标准操作的“早期”调度——在 InstallInitialize 操作之前或之后立即，您可以删除文件而不必担心组件规则违规。

The MSI SDK notes that scheduling RemoveExistingProducts early is “inefficient” because the files that are same between the two product versions are removed and then reinstalled. But that inefficiency is what lets you remove files and components.
MSI SDK 指出，提前安排 RemoveExistingProducts 是“低效的”，因为两个产品版本之间相同的文件会被删除然后重新安装。但这种低效性正是让你删除文件和组件的原因。

Authoring patches​ 编写补丁

Patch authoring starts with the Patch element. Attributes supply some metadata about the patch and child elements define the cabinet containing updated files and the target and update packages. The target and update packages can be specified as .wixpdb or as .msi files.
Patch 编写从 Patch 元素开始。属性提供了一些关于 patch 的元数据，子元素定义了包含更新文件的柜子以及目标和更新包。目标和更新包可以指定为.wixpdb 或.msi 文件。

Authoring patches using .wixpdb files​
使用 .wixpdb 文件编写补丁

.wixpdb files are produced during the WiX build and, like .pdb files from your favorite Visual Studio compiler, contain symbol information produced during the build process beyond what's recorded in the final output (like an .msi file). For patches, one bit of that extra information is the paths that were used when building the target and update packages to choose the payload files. When use you bind paths, WiX can use path information from the .wixpdb files to locate files when building the patch content.
.wixpdb 文件在 WiX 构建过程中生成，就像您最喜欢的 Visual Studio 编译器生成的 .pdb 文件一样，包含构建过程中生成的符号信息，这些信息超出了最终输出（如 .msi 文件）中记录的内容。对于补丁，这些额外信息的一部分是用于构建目标和更新包以选择有效负载文件的路径。当您使用绑定路径时，WiX 可以使用 .wixpdb 文件中的路径信息在构建 patch 内容时定位文件。

Here's what authoring for a patch might look like:
这是编写 patch 的示例：

<Wix xmlns="http://wixtoolset.org/schemas/v4/wxs">
  <Patch
    AllowRemoval="yes"
    DisplayName="Product Patch v$(Version)"
    Description="Product Patch v$(Version)"
    MoreInfoURL="https://wixtoolset.org/"
    Manufacturer="WiX Toolset"
    Classification="Update">

    <Media Id="1" Cabinet="patch.cab">
      <PatchBaseline
        Id="RTM"
        BaselineFile="PackageV1.wixpdb"
        UpdateFile="PackageV2.wixpdb" />
    </Media>

    <PatchFamily Id="SequenceFamily" Version="$(Version)" />
  </Patch>
</Wix>

Building it would look like this:
构建它看起来像这样：

wix build -define Version=2.0 -bindpath path\to\V1PackageOutput -bindpath path\to\V2PackageOutput -bindpath:target path\to\V1PackagePayloads -bindpath:update path\to\V2PackagePayloads -out path\to\patch.msp Patch.wxs

The following table describes the switches used for building the patch:
下表描述了用于构建patch的开关：

SwitchDescription
-definePasses the patch version into the patch authoring.
将patch版本传递到patch创作中。
-bindpath path\to\V1PackageOutputProvides a bind path to find the target package.
提供一个绑定路径来查找目标包。
-bindpath path\to\V2PackageOutputProvides a bind path to find the update package.
提供一个绑定路径来查找更新包。
-bindpath:target path\to\V1PackagePayloadsTells WiX to use the specified path when locating files from target .wixpdb.
告诉 WiX 在定位目标.wixpdb 文件时使用指定的路径。
-bindpath:update path\to\V2PackagePayloadsTells WiX to use the specified path when locating files from update .wixpdb.
告诉 WiX 在定位更新.wixpdb 文件时使用指定的路径。
-out path\to\patch.mspSpecify the output is a patch.
指定输出是一个patch。

Authoring patches using .msi files​
使用.msi 文件编写补丁

You can also use .msi files as the target and update packages. The primary advantage of using .msi files over .wixpdb files is that the .msi files are the source of both the changes between the target and update packages but also of the payload files themselves. So rather than supplying .wixpdb files and paths for the payload files, you need to supply just the .msi files. The primary disadvantage of using .msi files is that WiX needs to extract files from the .msi packages, which typically increases the time needed to build the patch.
您还可以使用.msi 文件作为目标和更新包。使用.msi 文件而不是.wixpdb 文件的主要优点是.msi 文件不仅是目标和更新包之间变化的来源，也是有效负载文件本身的来源。因此，与其提供.wixpdb 文件和有效负载文件的路径，不如只提供.msi 文件。使用.msi 文件的主要缺点是 WiX 需要从.msi 包中提取文件，这通常会增加构建patch所需的时间。

The patch authoring itself is identical other than file paths:
除了文件路径外，patch的编写本身是相同的：

<PatchBaseline
  Id="RTM"
  BaselineFile="PackageV1.msi"
  UpdateFile="PackageV2.msi" />

Building it would look like this:
构建它看起来像这样：

wix build -define Version=2.0 -bindpath path\to\V1PackageOutput -bindpath path\to\V2PackageOutput  -out PatchV1V2.msp Patch.wxs

The following table describes the switches used for building the patch:
下表描述了用于构建patch的开关：

SwitchDescription
-definePasses the patch version into the patch authoring.
将patch版本传递到patch创作中。
-bindpath path\to\V1PackageOutputProvides a bind path to find the target package.
提供一个绑定路径以找到目标包。
-bindpath path\to\V2PackageOutputProvides a bind path to find the update package.
提供一个绑定路径以找到更新包。
-out path\to\patch.mspSpecify the output is a patch.
指定输出是一个patch。
