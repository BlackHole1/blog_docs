Validation 验证

Package validation checks the content of MSI packages to, among other things, ensure the internal database structure follows (at least some of) the rules for MSI packages.
包验证检查 MSI 包的内容，以确保内部数据库结构遵循（至少一些）MSI 包的规则。

Building a package using MSBuild automatically runs validation using the stock MSI SDK ICEs. To run validation when using the Wix.exe .NET tool, use the wix msi validate subcommand.
使用 MSBuild 构建包时会自动运行验证，使用的是标准 MSI SDK ICEs。要在使用 Wix.exe .NET 工具时运行验证，请使用 wix msi validate 子命令。

Errors running validation​
运行验证时出错

Because the stock MSI ICEs are implemented as custom actions that are executed after merging them into an MSI package being built, the user running validation must be able to run Windows Installer. That works fine on a typical developer machine. However, non-interactive, non-administrator system accounts typically cannot. (System accounts that are administrators typically can.) Unfortunately, non-interactive system accounts are commonly used on continuous integration build machines. So if validation succeeds on your machine but fails during a CI build, you've run into this problem.
由于股票 MSI ICEs 作为自定义操作实现，在合并到正在构建的 MSI 包后执行，运行验证的用户必须能够运行 Windows Installer。这在典型的开发者机器上运行良好。然而，非交互式、非管理员系统帐户通常不能。（作为管理员的系统帐户通常可以。）不幸的是，非交互式系统帐户通常用于持续集成构建机器。因此，如果验证在您的机器上成功但在 CI 构建期间失败，您就遇到了这个问题。

Anecdotal evidence suggests that typical hosted CI build machines use service accounts that are incompatible with validation and because the hosted machines cannot be configured, there is no workaround to run validation during the build. The only option is to suppress validation by setting <SuppressValidation>true</SuppressValidation> in your WiX MSBuild project.
轶事证据表明，典型的托管 CI 构建机器使用与验证不兼容的服务帐户，并且由于托管机器无法配置，因此在构建期间没有运行验证的解决方法。唯一的选择是在您的 WiX MSBuild 项目中设置 <SuppressValidation>true</SuppressValidation> 来抑制验证。

Self-hosted CI build machines can run validation if the service account is LocalSystem or another administrator user.
如果服务帐户是 LocalSystem 或其他管理员用户，自托管 CI 构建机器可以运行验证。
