About 关于

The WiX Toolset lets developers create installers for Windows Installer, the Windows installation engine.
WiX 工具集让开发人员为 Windows Installer（Windows 安装引擎）创建安装程序。

  • The core of WiX is a set of build tools that build Windows Installer packages using the same build concepts as the rest of your product: source code is compiled and then linked to create executables; in this case .exe setup bundles, .msi installation packages, .msm merge modules, and .msp patches. The WiX command-line build tools work with any automated build system. Also, MSBuild is supported from the command line, Visual Studio, and common CI/CD build systems like GitHub Actions.
    WiX 的核心是一组构建工具，这些工具使用与产品其余部分相同的构建概念来构建 Windows Installer 包：源代码被编译然后链接以创建可执行文件；在这种情况下，是.exe 安装包、.msi 安装包、.msm 合并模块和.msp 补丁。WiX 命令行构建工具可以与任何自动化构建系统一起工作。此外，MSBuild 支持从命令行、Visual Studio 和常见的 CI/CD 构建系统（如 GitHub Actions）。

  • WiX includes several extensions that offer functionality beyond that of Windows Installer. For example, WiX can install IIS web sites, create SQL Server databases, and register exceptions in the Windows Firewall, among others.
    WiX 包含几个扩展，提供超出 Windows Installer 功能的功能。例如，WiX 可以安装 IIS 网站、创建 SQL Server 数据库，并在 Windows 防火墙中注册例外等。

  • With Burn, the WiX bootstrapper, you can create setup bundles that install prerequisites like the .NET Framework and other runtimes along with your own product. Burn lets you download packages or combine them into a single downloadable .exe.
    使用 Burn，WiX 引导程序，您可以创建安装包，安装 .NET Framework 和其他运行时等先决条件以及您自己的产品。Burn 允许您下载包或将它们组合成一个可下载的 .exe 文件。

  • The WiX SDK includes managed and native libraries that make it easier to write code that works with Windows Installer, including custom actions in both C# and C++.
    WiX SDK 包括托管和本机库，使编写与 Windows Installer 一起工作的代码更容易，包括 C# 和 C++ 中的自定义操作。

You can also follow via Twitter at @wixtoolset
您还可以通过 Twitter 关注@wixtoolset

System requirements​ 系统要求

Running packages built with WiX​
运行使用 WiX 构建的软件包

In general, packages you build with WiX will work on Windows 7 or later. Code that you introduce -- for example, custom actions or bootstrapper applications -- can require later versions of Windows.
一般来说，您使用 WiX 构建的软件包可以在 Windows 7 或更高版本上运行。您引入的代码——例如，自定义操作或引导程序应用程序——可能需要更高版本的 Windows。

Building packages with WiX​
使用 WiX 构建软件包

To use WiX as a .NET tool or as an MSBuild SDK via dotnet build, you must have a .NET 6 SDK installed. See its system requirements and download here.
要通过 dotnet build 将 WiX 用作.NET 工具或 MSBuild SDK，您必须安装.NET 6 SDK。请参阅其系统要求并在此处下载。

To use WiX as an MSBuild SDK via msbuild, you must have .NET Framework 4.7.2 or later installed. WiX runs on ARM64 systems as ARM64
要通过 msbuild 将 WiX 用作 MSBuild SDK，您必须安装.NET Framework 4.7.2 或更高版本。WiX 在 ARM64 系统上作为 ARM64 运行。

Governance​ 治理

This project is led and managed by a benevolent dictator. The benevolent dictator is responsible for the general strategic direction in addition to the day-to-day maintenance of the project. The community guides the decisions of the benevolent dictator through active engagement and contribution.
该项目由一位仁慈的独裁者领导和管理。仁慈的独裁者负责总体战略方向以及项目的日常维护。社区通过积极参与和贡献来指导仁慈独裁者的决策。

This project has also adopted the code of conduct defined by the Contributor Covenant to clarify expected behavior in our community. For more information see the .NET Foundation Code of Conduct.
该项目还采用了贡献者公约定义的行为准则，以明确我们社区的预期行为。有关更多信息，请参见 .NET 基金会行为准则。

Roles and Responsibilities​
角色和职责

Benevolent dictator (project leads)​
仁慈的独裁者（项目负责人）

In the WiX Toolset the role of Benevolent Dictator (project leads) is shared between Rob Mensching and Bob Arnson. The project leads are expected to understand the community as a whole and strive to satisfy as many conflicting needs as possible, while ensuring that the project survives in the long term.
在 WiX 工具集中，仁慈独裁者（项目负责人）的角色由 Rob Mensching 和 Bob Arnson 共同担任。项目负责人需要了解整个社区，并努力满足尽可能多的冲突需求，同时确保项目在长期内得以生存。

In many ways, the role of the benevolent dictator is less about dictatorship and more about diplomacy. The key is to ensure that, as the project expands, the right people are given influence over it and the community rallies behind the vision of the project leads.
在许多方面，仁慈独裁者的角色与其说是独裁，不如说是外交。关键在于确保随着项目的扩展，合适的人能够对其产生影响，并且社区能够支持项目负责人的愿景。

Additionally, .NET Foundation staff considers the project leads as the primary point of contact or first point of contact for the project for purposes of business operations including domain registrations, and technical services (e.g. code-signing).
此外，.NET 基金会的工作人员将项目负责人视为项目的主要联系人或首要联系人，负责包括域名注册和技术服务（如代码签名）在内的业务操作。

Committers​ 提交者

Committers are contributors who have made sustained valuable contributions to the project and are now relied upon to both write code directly to the repository. In many cases they are programmers but it is also possible that they contribute in a different role. Typically, a committer will focus on a specific aspect of the project, and will bring a level of expertise and understanding that earns them the respect of the community and the project lead. The role of committer is not an official one; it is simply a position that influential members of the community will find themselves in as the project lead looks to them for guidance and support.
提交者是那些对项目做出了持续有价值贡献的贡献者，现在他们被依赖于直接向代码库编写代码。在许多情况下，他们是程序员，但也有可能以不同的角色做出贡献。通常，提交者会专注于项目的特定方面，并带来一种专业知识和理解，赢得社区和项目负责人的尊重。提交者的角色不是官方的；这只是社区中有影响力的成员在项目负责人寻求他们的指导和支持时会发现自己处于这种位置。

Committers have no authority over the overall direction of the project. However, they do have the ear of the project leads. It is a committer's job to ensure that the lead is aware of the community's needs and collective objectives, and to help develop or elicit appropriate contributions to the project. Often, committers are given informal control over their specific areas of responsibility, and are assigned rights to directly modify certain areas of the source code. That is, although committers do not have explicit decision-making authority, they will often find that their actions are synonymous with the decisions made by the leads.
提交者对项目的整体方向没有权威。然而，他们确实能听到项目负责人的意见。提交者的工作是确保负责人了解社区的需求和集体目标，并帮助开发或引出适当的项目贡献。通常，提交者在其特定责任领域内被赋予非正式的控制权，并被分配直接修改某些源代码区域的权限。也就是说，尽管提交者没有明确的决策权，但他们的行动往往与负责人做出的决定同义。

How to become a Committer: Be a regular Contributor then be appointed by the Benevolent Dictator.
如何成为提交者：先成为常规贡献者，然后由仁慈的独裁者任命。

Contributors​ 贡献者

Contributors are community members who submit pull requests for the project. These pull requests may be a one-time occurrence or occur over time. The expectation is that that contributors will submit pull requests that are small at first. Ideally larger or more complex contributions are sent once the contributor has built confidence in the quality of their pull requests with the community.
贡献者是为项目提交拉取请求的社区成员。这些拉取请求可能是一次性的，也可能是长期的。期望贡献者首先提交小型的拉取请求。理想情况下，贡献者在社区中建立了对其拉取请求质量的信心后，再发送更大或更复杂的贡献。

Before a contributor's first pull request is put into the repository they must sign a Contribution License Agreement. The pull request can be submitted and discussed but it cannot be committed to the repository without the appropriate paperwork in place.
在贡献者的第一个拉取请求被放入存储库之前，他们必须签署贡献许可协议。拉取请求可以提交和讨论，但在没有适当的文书工作到位的情况下，不能提交到存储库。

How to become a Contributor: Participate in the project as a developer.
如何成为贡献者：以开发者的身份参与项目。

Users​ 用户

Users are community members who have a need for the project. They are the most important members of the community: without them, the project would have no purpose. Anyone can be a user; there are no specific requirements.
用户是对项目有需求的社区成员。他们是社区中最重要的成员：没有他们，项目将毫无意义。任何人都可以成为用户；没有具体的要求。

Users should be encouraged to participate in the life of the project and the community as much as possible. User contributions enable the project team to ensure that they are satisfying the needs of those users. Common user activities include (but are not limited to):
应鼓励用户尽可能多地参与项目和社区的生活。用户的贡献使项目团队能够确保满足这些用户的需求。常见的用户活动包括（但不限于）：

  • Advocating the use of the project
    倡导使用该项目
  • Informing developers of the project strength and weaknesses from a new user's perspective
    从新用户的角度告知开发人员项目的优缺点
  • Providing moral support (a 'thank you' goes a long way)
    提供道德支持（一个“谢谢”会有很大帮助）
  • Writing documentation 编写文档
  • Filing bug reports and feature requests
    提交错误报告和功能请求
  • Participating on the mailing lists and discussion forum
    参与邮件列表和讨论论坛

Users who continue to engage with the project and its community will often find themselves becoming more and more involved. Such users may then go on to become Contributors, as described above.
继续与项目及其社区互动的用户通常会发现自己越来越多地参与其中。这些用户可能会成为贡献者，如上所述。

How to become a User: Use the WiX toolset and participate on the mailing lists and discussion forum.
如何成为用户：使用 WiX 工具集并参与邮件列表和讨论论坛。

Contribution License Agreement​
贡献许可协议

The WiX Toolset copyright is held by the .NET Foundation. Before we can accept any contributions you must have a Contribution License Agreement on file. Fortunately the process is very easy.
WiX 工具集的版权由 .NET 基金会持有。在我们接受任何贡献之前，您必须有一份贡献许可协议在档。幸运的是，这个过程非常简单。

When you submit your first pull request, there will be a prompt to accept the CLA in the comments. After reading the CLA you can accept it by responding with a comment saying I have read the CLA Document and I hereby sign the CLA. This will only necessary for your first pull request.
当您提交您的第一个拉取请求时，评论中会提示您接受 CLA。阅读 CLA 后，您可以通过回复评论 I have read the CLA Document and I hereby sign the CLA 来接受它。这仅在您的第一个拉取请求时需要。

So go out there! Participate in design discussions. Get your code written. Test it. And test it some more. Post the pull request for code review. Then respond to the CLA prompt and your contribution will be ready to be merged into the WiX Toolset.
所以出去吧！参与设计讨论。编写你的代码。测试它。并且再测试几次。发布代码审查的拉取请求。然后响应 CLA 提示，你的贡献将准备好合并到 WiX 工具集中。

Happy coding! 编码愉快！
