Name 名称

systemd-sysext, systemd-sysext.service, systemd-confext, systemd-confext.service — Activates System Extension Images
systemd-sysext，systemd-sysext.service，systemd-confext，systemd-confext.service — 激活系统扩展镜像

Synopsis 概要

systemd-sysext [OPTIONS...] COMMAND
systemd-sysext [选项...] 命令

systemd-sysext.service

systemd-confext [OPTIONS...] COMMAND
systemd-confext [选项...] 命令

Description¶ 描述

systemd-sysext activates/deactivates system extension images. System extension images may – dynamically at runtime — extend the /usr/ and /opt/ directory hierarchies with additional files. This is particularly useful on immutable system images where a /usr/ and/or /opt/ hierarchy residing on a read-only file system shall be extended temporarily at runtime without making any persistent modifications.
systemd-sysext 激活/停用系统扩展映像。系统扩展映像可能会在运行时动态地扩展 /usr/ 和 /opt/ 目录层次结构，添加额外的文件。这在不可变系统映像上特别有用，其中位于只读文件系统上的 /usr/ 和/或 /opt/ 层次结构需要在运行时临时扩展，而不进行任何持久性修改。

System extension images should contain files and directories similar in fashion to regular operating system tree. When one or more system extension images are activated, their /usr/ and /opt/ hierarchies are combined via "overlayfs" with the same hierarchies of the host OS, and the host /usr/ and /opt/ overmounted with it ("merging"). When they are deactivated, the mount point is disassembled — again revealing the unmodified original host version of the hierarchy ("unmerging"). Merging thus makes the extension's resources suddenly appear below the /usr/ and /opt/ hierarchies as if they were included in the base OS image itself. Unmerging makes them disappear again, leaving in place only the files that were shipped with the base OS image itself.
系统扩展映像应包含类似于常规操作系统树的文件和目录。当一个或多个系统扩展映像被激活时，它们的 /usr/ 和 /opt/ 层次结构通过 " overlayfs " 与主机操作系统的相同层次结构合并，并且主机 /usr/ 和 /opt/ 与之覆盖（"合并"）。当它们被停用时，挂载点被拆卸 — 再次显示未经修改的原始主机版本的层次结构（"拆分"）。因此合并使得扩展的资源突然出现在 /usr/ 和 /opt/ 层次结构下，就好像它们包含在基本操作系统映像中一样。拆分使它们再次消失，只留下与基本操作系统映像一起提供的文件。

Files and directories contained in the extension images outside of the /usr/ and /opt/ hierarchies are not merged, and hence have no effect when included in a system extension image. In particular, files in the /etc/ and /var/ included in a system extension image will not appear in the respective hierarchies after activation.
扩展图像中包含的文件和目录位于 /usr/ 和 /opt/ 层次结构之外，不会合并，因此在包含在系统扩展图像中时不会产生任何影响。特别是，在系统扩展图像中包含的 /etc/ 和 /var/ 中的文件在激活后不会出现在相应的层次结构中。

System extension images are strictly read-only, and the host /usr/ and /opt/ hierarchies become read-only too while they are activated.
系统扩展图像是严格只读的，主机 /usr/ 和 /opt/ 层次结构在激活时也变为只读。

System extensions are supposed to be purely additive, i.e. they are supposed to include only files that do not exist in the underlying basic OS image. However, the underlying mechanism (overlayfs) also allows overlaying or removing files, but it is recommended not to make use of this.
系统扩展应该是纯粹的增量，即它们应该只包含在基本操作系统映像中不存在的文件。然而，底层机制（overlayfs）也允许覆盖或删除文件，但建议不要使用这种功能。

System extension images may be provided in the following formats:
系统扩展图像可以以以下格式提供：

  1. Plain directories or btrfs subvolumes containing the OS tree
    包含操作系统树的普通目录或 btrfs 子卷

  2. Disk images with a GPT disk label, following the Discoverable Partitions Specification
    带有 GPT 磁盘标签的磁盘映像，遵循可发现分区规范

  3. Disk images lacking a partition table, with a naked Linux file system (e.g. erofs, squashfs or ext4)
    磁盘镜像缺少分区表，具有裸的 Linux 文件系统（例如 erofs、squashfs 或 ext4）

These image formats are the same ones that systemd-nspawn(1) supports via its --directory=/--image= switches and those that the service manager supports via RootDirectory=/RootImage=. Similar to them they may optionally carry Verity authentication information.
这些图像格式与 systemd-nspawn(1) 通过其 --directory= / --image= 开关支持的格式相同，以及服务管理器通过 RootDirectory= / RootImage= 支持的格式相同。与它们类似，它们可以选择性地携带 Verity 认证信息。

System extensions are searched for in the directories /etc/extensions/, /run/extensions/ and /var/lib/extensions/. The first two listed directories are not suitable for carrying large binary images, however are still useful for carrying symlinks to them. The primary place for installing system extensions is /var/lib/extensions/. Any directories found in these search directories are considered directory based extension images; any files with the .raw suffix are considered disk image based extension images. When invoked in the initrd, the additional directory /.extra/sysext/ is included in the directories that are searched for extension images. Note however, that by default a tighter image policy applies to images found there, though, see below. This directory is populated by systemd-stub(7) with extension images found in the system's EFI System Partition.
系统扩展在目录 /etc/extensions/ 、 /run/extensions/ 和 /var/lib/extensions/ 中搜索。列出的前两个目录不适合携带大型二进制图像，但仍然适用于携带到它们的符号链接。安装系统扩展的主要位置是 /var/lib/extensions/ 。在 initrd 中调用时，附加目录 /.extra/sysext/ 包含在搜索扩展图像的目录中。但是请注意，默认情况下，对在那里找到的图像应用更严格的策略，详见下文。该目录由 systemd-stub(7) 使用在系统的 EFI 系统分区中找到的扩展图像进行填充。

During boot OS extension images are activated automatically, if the systemd-sysext.service is enabled. Note that this service runs only after the underlying file systems where system extensions may be located have been mounted. This means they are not suitable for shipping resources that are processed by subsystems running in earliest boot. Specifically, OS extension images are not suitable for shipping system services or systemd-sysusers(8) definitions. See the Portable Services page for a simple mechanism for shipping system services in disk images, in a similar fashion to OS extensions. Note the different isolation on these two mechanisms: while system extension directly extend the underlying OS image with additional files that appear in a way very similar to as if they were shipped in the OS image itself and thus imply no security isolation, portable services imply service level sandboxing in one way or another. The systemd-sysext.service service is guaranteed to finish start-up before basic.target is reached; i.e. at the time regular services initialize (those which do not use DefaultDependencies=no), the files and directories system extensions provide are available in /usr/ and /opt/ and may be accessed.
在启动时，如果启用了 systemd-sysext.service ，操作系统扩展映像会自动激活。请注意，此服务仅在挂载了系统扩展可能位于的底层文件系统之后才运行。这意味着它们不适合用于由最早启动的子系统处理的资源。具体来说，操作系统扩展映像不适合用于提供系统服务或 systemd-sysusers(8)定义。请参阅便携服务页面，了解一种类似于操作系统扩展的方式在磁盘映像中提供系统服务的简单机制。请注意这两种机制的不同隔离：系统扩展直接通过附加文件扩展底层操作系统映像，这些文件看起来与它们似乎是在操作系统映像本身中提供的文件非常相似，因此不涉及安全隔离，而便携服务则意味着以某种方式进行服务级沙箱隔离。 systemd-sysext.service 服务保证在达到 basic.target 之前完成启动；也就是说，在常规服务初始化时（不使用 DefaultDependencies=no 的服务），系统扩展提供的文件和目录将在 /usr/ 和 /opt/ 中可用并可访问。

Note that there is no concept of enabling/disabling installed system extension images: all installed extension images are automatically activated at boot. However, you can place an empty directory named like the extension (no .raw) in /etc/extensions/ to "mask" an extension with the same name in a system folder with lower precedence.
请注意，没有启用/禁用已安装的系统扩展图像的概念：所有已安装的扩展图像在启动时会自动激活。但是，您可以在 /etc/extensions/ 中放置一个命名类似于该扩展的空目录（不包括 .raw ），以“掩盖”具有较低优先级的系统文件夹中同名的扩展。

A simple mechanism for version compatibility is enforced: a system extension image must carry a /usr/lib/extension-release.d/extension-release.NAME file, which must match its image name, that is compared with the host os-release file: the contained ID= fields have to match unless "_any" is set for the extension. If the extension ID= is not "_any", the SYSEXT_LEVEL= field (if defined) has to match. If the latter is not defined, the VERSION_ID= field has to match instead. If the extension defines the ARCHITECTURE= field and the value is not "_any" it has to match the kernel's architecture reported by uname(2) but the used architecture identifiers are the same as for ConditionArchitecture= described in systemd.unit(5). EXTENSION_RELOAD_MANAGER= can be set to 1 if the extension requires a service manager reload after application of the extension. Note that the for the reasons mentioned earlier: Portable Services remain the recommended way to ship system services. System extensions should not ship a /usr/lib/os-release file (as that would be merged into the host /usr/ tree, overriding the host OS version data, which is not desirable). The extension-release file follows the same format and semantics, and carries the same content, as the os-release file of the OS, but it describes the resources carried in the extension image.
版本兼容性的简单机制是强制执行的：系统扩展镜像必须携带一个 /usr/lib/extension-release.d/extension-release.NAME 文件，该文件必须与其镜像名称匹配，与主机 os-release 文件进行比较：包含的 ID= 字段必须匹配，除非为扩展设置了“ _any ”。如果扩展 ID= 不是“ _any ”，则 SYSEXT_LEVEL= 字段（如果已定义）必须匹配。如果后者未定义，则 VERSION_ID= 字段必须匹配。如果扩展定义了 ARCHITECTURE= 字段且值不是“ _any ”，则必须与 uname(2)报告的内核架构匹配，但所使用的架构标识与 systemd.unit(5)中描述的相同。如果扩展需要应用后重新加载服务管理器，则 EXTENSION_RELOAD_MANAGER= 可以设置为 1。请注意，出于前述原因：便携式服务仍然是推荐的系统服务交付方式。系统扩展不应该包含 /usr/lib/os-release 文件（因为这将合并到主机 /usr/ 树中，覆盖主机操作系统版本数据，这是不可取的）。 extension-release 文件遵循相同的格式和语义，并携带与 OS 的 os-release 文件相同的内容，但它描述了扩展镜像中携带的资源。

The systemd-confext concept follows the same principle as the systemd-sysext(1) functionality but instead of working on /usr and /opt, confext will extend only /etc. Files and directories contained in the confext images outside of the /etc/ hierarchy are not merged, and hence have no effect when included in the image. Formats for these images are of the same as sysext images. The merged hierarchy will be mounted with "nosuid" and (if not disabled via --noexec=false) "noexec".
systemd-confext 概念遵循与 systemd-sysext(1)功能相同的原则，但与 /usr 和 /opt 不同，confext 将仅扩展 /etc 。在 confext 镜像中包含的文件和目录不会与 /etc/ 层次结构之外的内容合并，因此在图像中包含时不会产生影响。这些镜像的格式与 sysext 镜像相同。合并的层次结构将使用“ nosuid ”挂载，并且（如果未通过 --noexec=false 禁用）将使用“ noexec ”。

Confexts are looked for in the directories /run/confexts/, /var/lib/confexts/, /usr/lib/confexts/ and /usr/local/lib/confexts/. The first listed directory is not suitable for carrying large binary images, however is still useful for carrying symlinks to them. The primary place for installing configuration extensions is /var/lib/confexts/. Any directories found in these search directories are considered directory based confext images; any files with the .raw suffix are considered disk image based confext images.
在目录 /run/confexts/ 、 /var/lib/confexts/ 、 /usr/lib/confexts/ 和 /usr/local/lib/confexts/ 中查找 Confexts。列出的第一个目录不适合携带大型二进制图像，但仍然适用于携带到它们的符号链接。安装配置扩展的主要位置是 /var/lib/confexts/ 。在这些搜索目录中找到的任何目录都被视为基于目录的 confext 镜像；任何带有 .raw 后缀的文件都被视为基于磁盘镜像的 confext 镜像。

Again, just like sysext images, the confext images will contain a /etc/extension-release.d/extension-release.NAME file, which must match the image name (with the usual escape hatch of the user.extension-release.strict xattr(7)), and again with content being one or more of ID=, VERSION_ID=, and CONFEXT_LEVEL. Confext images will then be checked and matched against the base OS layer.
就像 sysext 图像一样，confext 图像将包含一个 /etc/extension-release.d/extension-release.NAME 文件，该文件必须与图像名称匹配（通常使用 user.extension-release.strict xattr(7)的转义方式），并且内容可以是一个或多个 ID= ， VERSION_ID= 和 CONFEXT_LEVEL 。然后将检查 confext 图像并与基本 OS 层进行匹配。

Uses¶ 用途 ¶

The primary use case for system images are immutable environments where debugging and development tools shall optionally be made available, but not included in the immutable base OS image itself (e.g. strace(1) and gdb(1) shall be an optionally installable addition in order to make debugging/development easier). System extension images should not be misunderstood as a generic software packaging framework, as no dependency scheme is available: system extensions should carry all files they need themselves, except for those already shipped in the underlying host system image. Typically, system extension images are built at the same time as the base OS image — within the same build system.
系统图像的主要用例是不可变环境，其中调试和开发工具可以选择性地提供，但不包含在不可变基本 OS 图像本身中（例如，strace(1)和 gdb(1)应作为可选安装的附加项，以便更轻松地进行调试/开发）。不应将系统扩展图像误解为通用软件打包框架，因为没有可用的依赖方案：系统扩展应携带它们自己需要的所有文件，除了已在底层主机系统图像中提供的文件。通常，系统扩展图像与基本 OS 图像同时构建 - 在同一构建系统中。

Another use case for the system extension concept is temporarily overriding OS supplied resources with newer ones, for example to install a locally compiled development version of some low-level component over the immutable OS image without doing a full OS rebuild or modifying the nominally immutable image. (e.g. "install" a locally built package with DESTDIR=/var/lib/extensions/mytest make install && systemd-sysext refresh, making it available in /usr/ as if it was installed in the OS image itself.) This case works regardless if the underlying host /usr/ is managed as immutable disk image or is a traditional package manager controlled (i.e. writable) tree.
系统扩展概念的另一个用例是临时使用更新的资源覆盖操作系统提供的资源，例如，在不进行完整操作系统重建或修改名义上不可变的映像的情况下，安装本地编译的开发版本的某个低级组件。 （例如，使用 DESTDIR=/var/lib/extensions/mytest make install && systemd-sysext refresh 安装本地构建的软件包，使其在 /usr/ 中可用，就像它是在操作系统映像本身中安装的一样。） 无论基础主机 /usr/ 是作为不可变磁盘映像还是作为传统软件包管理器控制（即可写）树进行管理，此用例均有效。

For the confext case, the OSConfig project aims to perform runtime reconfiguration of OS services. Sometimes, there is a need to swap certain configuration parameter values or restart only a specific service without deployment of new code or a complete OS deployment. In other words, we want to be able to tie the most frequently configured options to runtime updateable flags that can be changed without a system reboot. This will help reduce servicing times when there is a need for changing the OS configuration.
对于 confext 情况，OSConfig 项目旨在执行操作系统服务的运行时重新配置。 有时，需要交换某些配置参数值或仅重新启动特定服务，而无需部署新代码或完整的操作系统部署。 换句话说，我们希望能够将最常配置的选项与可在运行时更新的标志相结合，而无需系统重新启动即可更改这些选项。 当需要更改操作系统配置时，这将有助于减少维护时间。

Commands¶ 命令 ¶

The following commands are understood by both the sysext and confext concepts:
以下命令适用于 sysext 和 confext 概念：

status¶

When invoked without any command verb, or when status is specified the current merge status is shown, separately (for both /usr/ and /opt/ of sysext and for /etc/ of confext).
在不带任何命令动词或指定 status 时，显示当前合并状态，分别显示 sysext 和 confext 的 /usr/ 和 /opt/ ，以及 confext 的 /etc/ 。

Added in version 248. 在版本 248 中添加。

merge¶

Merges all currently installed system extension images into /usr/ and /opt/, by overmounting these hierarchies with an "overlayfs" file system combining the underlying hierarchies with those included in the extension images. This command will fail if the hierarchies are already merged. For confext, the merge happens into the /etc/ directory instead.
将所有当前安装的系统扩展映像合并到 /usr/ 和 /opt/ 中，通过使用“ overlayfs ”文件系统覆盖这些层次结构，将基础层次结构与扩展映像中包含的层次结构相结合。如果层次结构已经合并，则此命令将失败。对于 confext，合并将发生在 /etc/ 目录中。

Added in version 248. 在版本 248 中添加。

unmerge¶

Unmerges all currently installed system extension images from /usr/ and /opt/ for sysext and /etc/, for confext, by unmounting the "overlayfs" file systems created by merge prior.
从 /usr/ 和 /opt/ 分别卸载所有当前安装的系统扩展映像，对于 sysext 和 /etc/ ，对于 confext，通过卸载由 merge 在之前创建的“ overlayfs ”文件系统。

Added in version 248. 在版本 248 中添加。

refresh¶

A combination of unmerge and merge: if already mounted the existing "overlayfs" instance is unmounted temporarily, and then replaced by a new version. This command is useful after installing/removing system extension images, in order to update the "overlayfs" file system accordingly. If no system extensions are installed when this command is executed, the equivalent of unmerge is executed, without establishing any new "overlayfs" instance. Note that currently there's a brief moment where neither the old nor the new "overlayfs" file system is mounted. This implies that all resources supplied by a system extension will briefly disappear — even if it exists continuously during the refresh operation.
unmerge 和 merge 的组合：如果已挂载现有的“ overlayfs ”实例将被临时卸载，然后替换为新版本。此命令在安装/移除系统扩展映像后非常有用，以便相应地更新“ overlayfs ”文件系统。如果在执行此命令时未安装任何系统扩展，则执行相当于 unmerge 的操作，而不建立任何新的“ overlayfs ”实例。请注意，在刷新操作期间，旧的“ overlayfs ”文件系统和新的“ overlayfs ”文件系统都未挂载的瞬间。这意味着系统扩展提供的所有资源将在刷新操作期间短暂消失 — 即使在刷新操作期间持续存在。

Added in version 248. 在版本 248 中添加。

list¶

A brief list of installed extension images is shown.
显示已安装扩展映像的简要列表。

Added in version 248. 在版本 248 中添加。

Options¶ 选项

--root=¶

Operate relative to the specified root directory, i.e. establish the "overlayfs" mount not on the top-level host /usr/ and /opt/ hierarchies for sysext or /etc/ for confext, but below some specified root directory.
相对于指定的根目录操作，即在顶层主机 /usr/ 和 /opt/ 层次结构中建立 " overlayfs " 挂载点，用于 sysext 或 /etc/ 用于 confext，而不是在某个指定的根目录下方。

Added in version 248. 在版本 248 中添加。

--force¶

When merging system extensions into /usr/ and /opt/ for sysext and /etc/ for confext, ignore version incompatibilities, i.e. force merging regardless of whether the version information included in the images matches the host or not.
将系统扩展合并到 /usr/ 和 /opt/ 用于 sysext 和 /etc/ 用于 confext 时，忽略版本不兼容性，即强制合并，无论图像中包含的版本信息是否与主机匹配。

Added in version 248. 在版本 248 中添加。

--image-policy=policy¶

Takes an image policy string as argument, as per systemd.image-policy(7). The policy is enforced when operating on system extension disk images. If not specified defaults to "root=verity+signed+encrypted+unprotected+absent:usr=verity+signed+encrypted+unprotected+absent" for system extensions, i.e. only the root and /usr/ file systems in the image are used. For configuration extensions defaults to "root=verity+signed+encrypted+unprotected+absent". When run in the initrd and operating on a system extension image stored in the /.extra/sysext/ directory a slightly stricter policy is used by default: "root=signed+absent:usr=signed+absent", see above for details.
以图像策略字符串作为参数，如 systemd.image-policy(7)。在操作系统扩展磁盘映像时执行策略。如果未指定，默认为系统扩展的 " root=verity+signed+encrypted+unprotected+absent:usr=verity+signed+encrypted+unprotected+absent "，即仅使用图像中的根目录和 /usr/ 文件系统。对于配置扩展，默认为 " root=verity+signed+encrypted+unprotected+absent "。在 initrd 中运行并操作存储在 /.extra/sysext/ 目录中的系统扩展映像时，默认使用略严格的策略：" root=signed+absent:usr=signed+absent "，有关详细信息，请参阅上文。

Added in version 254. 在版本 254 中添加。

--noexec=BOOL¶

When merging configuration extensions into /etc/ the "MS_NOEXEC" mount flag is used by default. This option can be used to disable it.
当将配置扩展合并到 /etc/ 时，默认使用“ MS_NOEXEC ”挂载标志。可以使用此选项将其禁用。

Added in version 254. 在版本 254 中添加。

--no-reload¶

When used with merge, unmerge or refresh, do not reload daemon after executing the changes even if an extension that is applied requires a reload via the EXTENSION_RELOAD_MANAGER= set to 1.
与合并、取消合并或刷新一起使用时，在执行更改后不要重新加载守护程序，即使应用的扩展需要通过将 EXTENSION_RELOAD_MANAGER= 设置为 1 来重新加载。

--no-pager¶

Do not pipe output into a pager.
不要将输出导入分页器。

--no-legend¶

Do not print the legend, i.e. column headers and the footer with hints.
不打印图例，即列标题和带有提示的页脚。

--json=MODE¶

Shows output formatted as JSON. Expects one of "short" (for the shortest possible output without any redundant whitespace or line breaks), "pretty" (for a pretty version of the same, with indentation and line breaks) or "off" (to turn off JSON output, the default).
以 JSON 格式显示输出。期望其中之一为 " short "（以最短可能的输出形式，不包含任何多余的空格或换行符）、" pretty "（相同内容的漂亮版本，带有缩进和换行符）或 " off "（关闭 JSON 输出，即默认设置）。

Exit status¶ 退出状态。

On success, 0 is returned.
成功时返回 0。
