systemd.offline-updates — 在 systemd 中实现离线更新

Implementing Offline System Updates¶
实现离线系统更新 ¶

This man page describes how to implement "offline" system updates with systemd. By "offline" OS updates we mean package installations and updates that are run with the system booted into a special system update mode, in order to avoid problems related to conflicts of libraries and services that are currently running with those on disk. This document is inspired by this GNOME design whiteboard.
本手册描述了如何使用 systemd 实现“离线”系统更新。通过“离线”操作系统更新，我们指的是在系统引导到特殊的系统更新模式时运行的软件包安装和更新，以避免与当前正在运行的库和服务发生冲突的问题。本文档受到了 GNOME 设计白板的启发。

The logic: 逻辑：

  1. The package manager prepares system updates by downloading all (.rpm or .deb or whatever) packages to update off-line in a special directory /var/lib/system-update (or another directory of the package/upgrade manager's choice).
    包管理器通过下载所有（.rpm 或 .deb 或其他）要离线更新的软件包，准备系统更新到一个特殊目录 /var/lib/system-update （或包/升级管理器选择的另一个目录）。

  2. When the user OK'ed the update, the symlink /system-update or /etc/system-update is created that points to /var/lib/system-update (or wherever the directory with the upgrade files is located) and the system is rebooted. This symlink is in the root directory, since we need to check for it very early at boot, at a time where /var/ is not available yet.
    当用户确认更新后，创建一个符号链接 /system-update 或 /etc/system-update 指向 /var/lib/system-update （或包含升级文件的目录的位置），然后系统重新启动。这个符号链接位于根目录，因为我们需要在引导时非常早的时候检查它，此时 /var/ 还不可用。

  3. Very early in the new boot systemd-system-update-generator(8) checks whether /system-update or /etc/system-update exists. If so, it (temporarily and for this boot only) redirects (i.e. symlinks) default.target to system-update.target, a special target that pulls in the base system (i.e. sysinit.target, so that all file systems are mounted but little else) and the system update units.
    在新引导的早期，systemd-system-update-generator(8) 检查是否存在 /system-update 或 /etc/system-update 。如果是这样，它会（仅临时且仅限此引导）将 default.target 重定向（即创建符号链接）到 system-update.target ，一个特殊的目标，用于引入基本系统（即 sysinit.target ，以便挂载所有文件系统但几乎不做其他操作）和系统更新单元。

  4. The system now continues to boot into default.target, and thus into system-update.target. This target pulls in all system update units. Only one service should perform an update (see the next point), and all the other ones should exit cleanly with a "success" return code and without doing anything. Update services should be ordered after sysinit.target so that the update starts after all file systems have been mounted.
    系统现在继续引导进入 default.target ，因此进入 system-update.target 。此目标引入所有系统更新单元。只有一个服务应该执行更新（参见下一点），所有其他服务应该以“成功”返回代码干净地退出而不执行任何操作。更新服务应该在 sysinit.target 之后排序，以便在所有文件系统挂载后开始更新。

  5. As the first step, an update service should check if the /system-update or /etc/system-update symlink points to the location used by that update service. In case it does not exist or points to a different location, the service must exit without error. It is possible for multiple update services to be installed, and for multiple update services to be launched in parallel, and only the one that corresponds to the tool that created the symlink before reboot should perform any actions. It is unsafe to run multiple updates in parallel.
    作为第一步，更新服务应检查 /system-update 或 /etc/system-update 符号链接是否指向该更新服务使用的位置。如果不存在或指向不同位置，则服务必须无错误退出。可能安装多个更新服务，并且可以并行启动多个更新服务，只有在重启前创建符号链接的工具对应的服务才能执行任何操作。并行运行多个更新是不安全的。

  6. The update service should now do its job. If applicable and possible, it should create a file system snapshot, then install all packages. After completion (regardless whether the update succeeded or failed) the machine must be rebooted, for example by calling systemctl reboot. In addition, on failure the script should revert to the old file system snapshot (without the symlink).
    更新服务现在应该开始工作。如果适用且可能，应创建文件系统快照，然后安装所有软件包。完成后（无论更新成功与否），必须重新启动计算机，例如通过调用 systemctl reboot。此外，在失败时，脚本应还原到旧的文件系统快照（不包括符号链接）。

  7. The update scripts should exit only after the update is finished. It is expected that the service which performs the update will cause the machine to reboot after it is done. If the system-update.target is successfully reached, i.e. all update services have run, and the /system-update or /etc/system-update symlink still exists, it will be removed and the machine rebooted as a safety measure.
    更新脚本应该在更新完成后才退出。预期执行更新的服务在完成后会导致机器重新启动。如果成功达到 system-update.target ，即所有更新服务已运行，并且 /system-update 或 /etc/system-update 符号链接仍然存在，则会将其删除并作为安全措施重新启动机器。

  8. After a reboot, now that the /system-update and /etc/system-update symlink is gone, the generator won't redirect default.target anymore and the system now boots into the default target again.
    重新启动后，现在 /system-update 和 /etc/system-update 符号链接已经消失，生成器将不再重定向 default.target ，系统现在会再次引导到默认目标。

Recommendations¶ 建议 ¶

  1. To make things a bit more robust we recommend hooking the update script into system-update.target via a .wants/ symlink in the distribution package, rather than depending on systemctl enable in the postinst scriptlets of your package. More specifically, for your update script create a .service file, without [Install] section, and then add a symlink like /usr/lib/systemd/system/system-update.target.wants/foobar.service → ../foobar.service to your package.
    为了使事情更加健壮，我们建议将更新脚本通过一个符号链接（symlink）挂钩到 system-update.target 中，而不是依赖于包的 postinst 脚本中的 systemctl enable。更具体地说，对于您的更新脚本，请创建一个没有 [Install] 部分的 .service 文件，然后将一个符号链接添加到您的包中，如 /usr/lib/systemd/system/system-update.target.wants/foobar.service → ../foobar.service 。

  2. Make sure to remove the /system-update and /etc/system-update symlinks as early as possible in the update script to avoid reboot loops in case the update fails.
    确保在更新脚本中尽早删除 /system-update 和 /etc/system-update 符号链接，以避免在更新失败时出现重启循环。

  3. Use FailureAction=reboot in the service file for your update script to ensure that a reboot is automatically triggered if the update fails. FailureAction= makes sure that the specified unit is activated if your script exits uncleanly (by non-zero error code, or signal/coredump). If your script succeeds you should trigger the reboot in your own code, for example by invoking logind's Reboot() call or calling systemctl reboot. See org.freedesktop.login1(5) for details about the logind D-Bus API.
    在您的更新脚本的服务文件中使用 FailureAction=reboot ，以确保如果更新失败，则会自动触发重启。 FailureAction= 确保如果您的脚本以不干净的方式退出（通过非零错误代码或信号/核心转储），则激活指定的单元。如果您的脚本成功，您应该在自己的代码中触发重启，例如通过调用 logind 的 Reboot() 调用或调用 systemctl reboot。有关 logind D-Bus API 的详细信息，请参阅 org.freedesktop.login1(5)。

  4. The update service should declare DefaultDependencies=no, Requires=sysinit.target, After=sysinit.target, After=system-update-pre.target, Before=system-update.target and explicitly pull in any other services it requires.
    更新服务应声明 DefaultDependencies=no ， Requires=sysinit.target ， After=sysinit.target ， After=system-update-pre.target ， Before=system-update.target 并明确拉取其所需的任何其他服务。

  5. It may be desirable to always run an auxiliary unit when booting into offline-updates mode, which itself does not install updates. To do this create a .service file with Wants=system-update-pre.target and Before=system-update-pre.target and add a symlink to that file under /usr/lib/systemd/system-update.target.wants .
    当启动到离线更新模式时，可能希望始终运行一个辅助单元，该单元本身不安装更新。要做到这一点，请创建一个带有 Wants=system-update-pre.target 和 Before=system-update-pre.target 的 .service 文件，并在 /usr/lib/systemd/system-update.target.wants 下创建到该文件的符号链接。
